## Decisions (locked at the 2026-07-03 gate)

- **D1 (24.6-b) Status taxonomy → option (b), derived display mapping; NO enum migration.** The Confluence story enumerates six display statuses: Active (Payment Pending) / Active (Paid In Full) / Cancelled (Full | Partial | No Refund | Never Paid). The Status Breakdowns tab settles semantics: Active pair is automatic (default; flips on full payment), Cancelled variants are *outcomes of the manual cancel flow* (per-milestone refund picker = exactly 24.9's per-installment primitive + 24.10's confirm flow). Qualifiers are pure arithmetic over 24.9's refund ledger: Never Paid (paid=0) / No Refund (paid>0, refunded=0) / Partial (0<refunded<paid) / Full (refunded=paid). API returns stored `status` + derived `status_display`. Rejected (a) enum extension: stores what is derivable, collides with 24.9's derived-refund-state decision. **BA note:** spec has no standalone "Refunded" top-level — a refund without cancel stays Active(…); stored `failed` maps to Active (Payment Pending). Installment display labels per sheet: succeeded→Paid, scheduled→Unpaid, failed→CC Failed, canceled→Voided, refunded→Refunded; chargeback trio = the "statuses no story produces" already ND-flagged in 24.8-l (this sheet is their source; its Jackie Questions tab shows disputes still unresolved at accounting-doc level).
- **D2 (24.6-h/m) Admin-checkout snapshot gaps → option (b), register to Cart Management team.** `checkout.service.ts` is **Suman Paul's Cart Management module** (created c0e1c46 2026-06-11, last touched bb94413 2026-06-29) — not ours to edit unilaterally (record-don't-own). 24.6 ships display + PATCH remedy; both missing write-sets go on one register row (OQ-1). Net effect admin-created vs exhibitor-created orders: empty billing address/phone (manually fixable via PATCH) and flat add-on list (not reconstructable post-hoc) — no crashes, money math identical.
- **D3 (24.6-j/t) Created-By/Source → option (a), read-time derivation; no new columns.** Chain: `sales_person_id` set → rep name ("Sales Person Sale") → cart `created_by_type=exhibitor` → **"Self-Service"** (story-mandated label) → cart `created_by_type=admin` → admin creator's name → no cart (subscription/ppl_addon) → order_type label. Covers 100% of rows, zero migration, avoids the cart-module write path. `OrderType` enum is the *billing* taxonomy — NOT the story's "Order Type (Sales Person Sale, Self-Service)"; do not conflate. Rep name resolved server-side via the existing `Order.salesPerson → User` relation + shared name-builder (same as the 24.1 list).
- **D4 (24.6-t) Universal-rep default = the derivation itself.** The story text supplies the default ("…the order source shall be 'Self-Service'"), so no house-account user, no backfill, no BA row. Reassignment (24.6-u) can attach a real rep to any order at any time.
- **D5 (24.6-i) Additional emails → option (a), `Order.additional_emails` Json (string array), edited via `PATCH :id`.** Child table rejected (no per-email metadata requirement anywhere). Whether these addresses get CC'd on *automated* sends is ONE consolidated BA question covering all four senders (OQ-2); worst case = one-line CC-append per sender, all still unbuilt.
- **D6 (24.6-l2) Per-item Company Contact → DEFERRED** (supersedes the gate's initial option-c lean). Code sweep: zero contact columns on OrderItem/CartItem anywhere; "Company Contact" appears exactly once on the whole Confluence page. Best-fit producer `OnsiteBoothContact` (exact 3-field match, keyed (company_id, show_id); orders ARE multi-show so values differ per item's show) gained its producer 2026-07-02 (exhibitor onsite-booth-contact module, Sushobhan Manna, SBE-1165 workstream) + edit story SBE-1171 merged 2026-07-03 — but display semantics belong to **SBE-1169 "View Onsite Boot Contact"** (Sprint 5, Ritam Mondal), and 13.3-w (the exhibitor twin) is Blocked on the same stories. Deferring keeps admin consistent with 13.3 and avoids pre-empting Sprint-5 semantics. Revisit trigger: SBE-1169 lands → the (company_id, show_id) join becomes a small additive read (OQ-3).
- **D7 (24.6-x, with 24.6-v) HubSpot sync → Needs clarification (BA); nothing built.** Create-once manual sync without `updateDeal` would create duplicate deals on second click. Full HubSpot inventory recorded in OQ-5; no owning epic exists.
- **D-z (24.6-z) Manual installment void → 24.6 OWNS it** (gate decision by Prantik). `POST /orders/:id/payments/:transactionId/void`: guard `status IN (scheduled, failed)` (sheet shows DELETE/VOID on Unpaid + CC Failed rows), sets PT → `canceled` + `next_retry_at: null` (same cascade shape 24.10 locked — retry cron can never resurrect a voided row), renders "Voided", key `orders.payments.void`, audited. Distinct from 24.8's scheduled-row *delete* and from 24.10's cancel-driven cascade.
- **24.6-r recipient contract (proposed, unratified — parked with the row):** default `[billing_email]`; optional `recipients[]` override (FE pre-populates billing + additional_emails + ad-hoc); 400 if no resolvable recipient; send by **template id** (sidesteps E&S known-issue #21 slug shadowing).

## Scope Finalization (2026-07-03, confirmed with Prantik)

- **No `/actions` endpoint (24.5-a):** the Quick Actions menu is front-end; the backend contribution is the per-action routes themselves, each gated by its own `@Permissions` key (the guard framework IS the enforcement). The feasibility doc's proposed `GET /orders/:id/actions` is dropped.
- **Booth Build (24.5-d) deferred** to its own future-sprint story, exactly as SBE-748 did on the cart side. The confirmed/signed status-mapping question travels with that story.
- **Agreement download = the SIGNED contract (24.5-e/f):** resolves the open signed-vs-unsigned question. The system convention is cart = unsigned draft (admin cart module renders live-cart PDF/DOCX with an empty signature block), order = signed contract (exhibitor `buildSignedBoothContractDocx` renders from the Order snapshot + `OrderAgreement` with the filled signature block, S3-cached via `OrderAgreement.signed_agreement_url`; admin ppl-order already ports the PPL twin at `GET /ppl-orders/:id/agreement`). 24.5-e = port the booth builder into the orders module (DOCX). 24.5-f = **net-new signed-PDF composition** (none exists anywhere) built here with admin's pdfmake service from the same order-snapshot context.
- **Invoice download is per-invoice (24.5-g):** booth orders get one Invoice row per successful installment, so the route mirrors the shipped PPL shape (`GET /orders/:id/invoices/:invoiceId/invoice`). The booth/product invoice PDF layout is net-new (no booth renderer exists), but the webhook already snapshots invoice line items as a typed tree expressly for it, and admin's `InvoiceDocumentService` (PPL port) provides the pattern + `Invoice.invoice_pdf_url` cache-first convention.
- **Additional users (24.5-h):** admin half delegates to the shipped SBE-748 `CartInviteUserService` flow, keyed off `order.company_id` — route mirrors the cart convention (`POST /orders/:id/invite-user`, one email per call), NOT the previously-proposed batch `additional-users` shape. The "access permission shall be set by the exhibitor" clause has no model anywhere (incl. the cart-management section) and is **flagged for BA removal from the story**.
- **Resend confirmation attaches the invoice PDF (24.5-i):** resolves the doc's second open question by in-repo precedent — admin PPL `resendOrderConfirmation` already attaches the latest invoice PDF via `getPdfBufferForAttachment` and throttles repeat sends. The booth port resends `booth_order_confirmation` with the latest invoice PDF rendered by 24.5-g's new renderer.
- **Resend portal password (24.5-j):** port of admin PPL `resendPortalPassword` (`admin_resend_portal_access` template) keyed on the order's company primary exhibitor.
- **Impersonation (24.5-k):** port of the shipped SBE-748 `CartImpersonateService` keyed off `order.company_id` (primary exhibitor → exhibitor JWTs → `impersonated_by` login log).
- **RBAC (24.5-l):** per-route permission keys seeded + role-granted + documented in Swagger (24.14 convention); no availability endpoint.
- **Logging (24.5-m):** delivered by construction — the ported services already write `impersonated_by`, activity/security logs, audit entries, and notification logs.

## Summary

This story asks for a per-order Quick Actions menu in Admin Panel > Order Management exposing: open Upsell / Onsite Sale / Booth Build workflows (Booth Build gated to confirmed/signed orders + a Booth Build role); Download Agreement (Word + PDF); Download Order Invoice (PDF); Update Additional Users (admin adds emails, exhibitor sets permissions); Resend Order Confirm + Invoice; Resend Portal Password; and permission-based Log in to Customer Portal impersonation - with backend enforcement and logging. There is NO order/order-management module in any repo, so the whole admin endpoint surface is net-new, but re-verification against the actual code shows the underlying primitives are far more complete than previously assessed and most already live IN admin-backend-api: agreement download in BOTH PDF (buildSimplePdf) and Word (.doc) is fully built in the admin cart module; a generic mailer.sendFromTemplate plus seeded order-confirmation/welcome templates back the resend actions; admin impersonation already keys off company_id (finds the company's primary exhibitor, mints exhibitor JWTs, writes impersonated_by); the role/permission framework and audit/login logging are proven. Building the net-new order endpoints on these built primitives is therefore Deliverable for 10 of 13 requirements. Only Update Additional Users stays Partial (admin-add-emails is feasible via the invite primitive, but the 'permissions set by exhibitor' piece has no built company-user permission model). Upsell and Onsite Sale workflows belong to other epics and are Out of Scope as link targets.

**Superseded at scope confirmation 2026-07-03 (see Scope Finalization below):** re-verification against current code found the substrate even stronger than this summary records — the SBE-748 cart quick actions shipped admin invite + impersonate services to port, the admin ppl-order module shipped signed-agreement/invoice/resend/portal-password ports to mirror, and the agreement decision moved from the unsigned cart generator to the SIGNED contract builders. 24.5-h flipped to Deliverable (exhibitor clause → BA-removal flag) and 24.5-d was deferred to the future Booth Build story.

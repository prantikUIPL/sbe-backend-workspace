## Context

**Why:** Admin Order Management (24.1–24.4, shipped) lists orders but offers no per-order actions — admins cannot hand a customer their signed contract or invoice, re-send a confirmation, invite portal users, or log into the portal to assist, without going through the cart/PPL modules (which cover carts and PPL orders only, not product orders). 24.6-s (Order Details agreement view/download) is blocked on this story's agreement routes.

**What:** Seven net-new routes on the shipped `OrdersModule`, all delegations/ports of proven admin code: a signed booth-contract DOCX port + a net-new signed-contract PDF composition (`OrderAgreementDocumentService`), a booth/product invoice PDF renderer (`OrderInvoiceDocumentService`, per-invoice, PPL mirror), and an `OrderActionsService` carrying invite-user, resend-confirmation (with invoice PDF attached), resend-portal-password, and impersonate — each `@Permissions`-gated and audited. **Zero schema changes in any repo** — every column these features need already exists.

## Design Decisions

### D1: Downloads serve the SIGNED contract; the cart generator stays the unsigned draft

Human decision 2026-07-03 (closes the doc's open question). Order = signed legal document (`OrderAgreement` snapshot: signer, signature image, `signed_at`, frozen `agreement_ids`); cart = pre-sign draft. The admin PPL quick action already serves the signed document — product orders follow the same convention.

Implication: the order agreement routes never read live Cart content; they render from `Order` + `OrderItem` + `OrderAgreement` + the frozen `Agreement` rows. 404 (`'No signed agreement found for this order'`, mirroring ppl-order) when the order has no `OrderAgreement` row.

### D2: `OrderAgreementDocumentService` — booth DOCX port, cache-first on the shared S3 column

Port `buildSignedBoothContractDocx` (+ `loadBoothContractOrderContext` and its booth table/signature/OOXML helpers) from exhibitor into `src/admin/orders/`, exactly as `admin/ppl-order/services/agreement-document.service.ts` ported the PPL twin ("Ported from exhibitor-backend's … so the document matches"). Replace the `exhibitorId` ownership check with the admin ownership-404 discipline (D7). Cache contract preserved byte-for-byte: reuse `OrderAgreement.signed_agreement_url` when fetchable; else render → upload → persist `signed_agreement_url` + `signed_agreement_generated_at`. Filename convention `SBE-Booth-Order-{order_number}-Signed.docx` (unchanged, so both sides serve the identical cached artifact).

Implication: the ported composition must stay content-identical to the exhibitor output — a divergent admin render would poison the shared cache. Any intentional layout change is out of scope here and would belong to both repos at once.

### D3: Signed PDF is net-new pdfmake composition; rendered per request, NO new cache column

No signed-PDF renderer exists in any repo, and `signed_agreement_url` caches DOCX only. Adding a PDF cache column means a migration + 4 sibling schema syncs for a marginal win. Decision: **no schema change** — `generate(orderId, 'pdf')` composes a pdfmake `TDocumentDefinitions` from the same loaded context (same tables/terms/filled-signature data as D2) and renders per request via the existing `PdfmakeService` (pure JS, already renders the cart agreement PDF). The document is immutable post-payment, so repeated renders are deterministic; render cost is acceptable for an admin-frequency action.

Implication: the context loader is format-agnostic (load once, branch to DOCX/PDF composers — same shape as the cart generator's `generate`). If PDF volume ever matters, a cache column is a later additive migration.

### D4: `OrderInvoiceDocumentService` — per-invoice PPL mirror with a booth/product layout branch

Route and semantics mirror the shipped PPL pair exactly: `GET /orders/:id/invoices/:invoiceId/invoice` returns `{ url }` (async-style download; renders → uploads → persists `Invoice.invoice_pdf_url`, cache-first; the cache column is DB-shared so any future exhibitor download reuses it), plus `getPdfBufferForAttachment(orderId, invoiceId)` for D6's email attachment. The layout branch is the genuinely new part: booth/product invoices render the typed `InvoiceLineItem` tree the webhook snapshots (nested "Included Items" under their parent booth line, fees split into the bottom totals area) instead of the PPL subscription/lead-credits layout. pdfkit, matching the ported renderer it extends.

Implication: invoice must belong to the order (`loadInvoice(orderId, invoiceId)` 404 discipline, as PPL does). Unpaid orders simply have no Invoice rows yet — the route 404s on a wrong/missing invoice id; no "generate from order lines" fallback (invoice semantics = persisted rows, decision recorded in the consolidation map).

### D5: Invite + impersonate are copy-adapted into the orders module, not cross-imported

`CartInviteUserService`/`CartImpersonateService` take `cart_id` and resolve the company through the cart; they are not exported from `CartModule`, and refactoring shipped SBE-748 code to extract a company-keyed core would touch another story's surface. Decision: **copy-adapt** (the established repo pattern — ppl-order copy-adapted exhibitor code) into an order-keyed `OrderActionsService`: same primary-holder resolution, same conflict/resurrect semantics, same `company_user_invitation` template and token TTL configuration key, same JWT minting + `impersonated_by` login-log + activity-log writes — keyed off `order.company_id`.

Implication: additive-only; cart module untouched. The duplication is deliberate and bounded (two flows), recorded here so a future shared-core refactor has both call sites named.

### D6: Resend pair ports the ppl-order actions stack; confirmation attaches the LATEST invoice PDF

Port `resendOrderConfirmation`/`resendPortalPassword` including `assertNotThrottled` (same notification-log throttle window) onto product orders: confirmation resends the seeded `booth_order_confirmation` template to the order's billing email (400 when absent, as PPL does), attaching the **latest** Invoice row's PDF via D4's `getPdfBufferForAttachment` (closes the doc's second open question by precedent). **When the order has no Invoice rows yet** (nothing paid), the resend sends **without attachment** rather than failing — the confirmation is still meaningful, and PPL's addon branch already establishes the no-attachment variant. Portal password sends `admin_resend_portal_access` to the company's primary exhibitor.

Implication: template variable mapping is the real work — mirror the variables the checkout-time `booth_order_confirmation` send populates (external-api webhook `:1850,2531` region) so the resend renders identically.

### D7: Ownership-404 discipline — product-only, plus per-route existence gates

All seven routes resolve the order with `findFirst({ where: { id, deleted_at: null, order_type: product } })` → `NotFoundException('Order not found')` otherwise (matches the shipped list + 24.14). On top: agreement routes 404 without an `OrderAgreement` row (D1); invoice route 404s when the invoice doesn't belong to the order (D4); invite/impersonate 400/404 when no linked company or no primary exhibitor (mirroring the cart services' messages).

### D8: RBAC — six new `orders.*` keys, Swagger-documented

`orders.agreement.read` (BOTH agreement formats — one document, two renderings), `orders.invoice.read`, `orders.invite-user`, `orders.resend-confirmation`, `orders.resend-portal-password`, `orders.impersonate`. Seeded in the permission.seeder orders block (after `orders.notes.update`, ids continue the sequence) with the route/method/url shape 24.14 used; granted in role.seeder to every array that grants `orders.list`. Every route's `@ApiOperation` description states its permission key + the `[SBE-24.5 — Quick Actions Per Order Item]` story header line (24.14/D6 convention).

### D9: Every action is order-entity audited

The ported cart/ppl code audits against `cart`/order-adjacent entities; here every route records against `AdminAuditEntityType.order` with `entityId = order.id`: agreement/invoice downloads (`note: 'Agreement document (DOCX) downloaded …'` style, best-effort non-blocking like the cart generator's `recordDownload`), invite (`'Portal user invited …'`), resends, and impersonation (which additionally writes the `impersonated_by` exhibitor login log + `logActivity`/`logSecurity`, ported as-is). This satisfies 24.5-m with no separate build item.

### Routes

| # | Method | Path | Status | Permission | Notes |
|---|--------|------|--------|------------|-------|
| 1 | GET | `/api/v1/orders/:id/agreement.docx` | 200 stream | `orders.agreement.read` | Signed booth contract DOCX; cache-first (shared S3 URL) |
| 2 | GET | `/api/v1/orders/:id/agreement.pdf` | 200 stream | `orders.agreement.read` | Signed contract PDF; rendered per request |
| 3 | GET | `/api/v1/orders/:id/invoices/:invoiceId/invoice` | 200 `{url}` | `orders.invoice.read` | Booth/product invoice PDF; cache-first on `invoice_pdf_url` |
| 4 | POST | `/api/v1/orders/:id/invite-user` | 201 | `orders.invite-user` | Body `{ email }`; one email per call |
| 5 | POST | `/api/v1/orders/:id/resend-confirmation` | 200 | `orders.resend-confirmation` | `booth_order_confirmation` + latest invoice PDF attached (or none if no invoice); throttled |
| 6 | POST | `/api/v1/orders/:id/resend-portal-password` | 200 | `orders.resend-portal-password` | `admin_resend_portal_access` to primary exhibitor; throttled |
| 7 | POST | `/api/v1/orders/:id/impersonate` | 200 | `orders.impersonate` | Returns exhibitor ACCESS/REFRESH tokens + customer summary (cart-route response shape) |

**Error mapping:** invalid ids/body → 400; missing/invalid JWT → 401; missing permission → 403; unknown / soft-deleted / non-product order → 404 `'Order not found'`; no `OrderAgreement` → 404 (routes 1–2); invoice not on order → 404 (route 3); no linked company / primary exhibitor → 400/404 with the cart services' messages (routes 4–7); throttled resend → 429/400 per the ppl-order port.

**Consumer responsibilities (24.6-s):** consume routes 1–2 as-is; no separate agreement route in 24.6.

### Step 1: Signed DOCX — `order-agreement-document.service.ts`

## Security Considerations

- **Impersonation is the sensitive surface:** permission-gated (`orders.impersonate`), tokens minted exactly as the shipped cart/ppl flows mint them, `impersonated_by` written to the exhibitor login log + activity/security logs (D9) — same accountability trail as SBE-748.
- Invite creates portal accounts: placeholder bcrypt password, invitation token TTL from configuration, conflict semantics prevent account takeover of existing users (ported checks).
- Signed documents and invoices are customer PII/legal artifacts: routes are admin-JWT + permission gated; S3 objects go through the existing UploadService pathing (same buckets/ACLs as the shipped flows).
- Resend throttle (D6) limits email-flood abuse from a compromised admin session.

# Story 24.2 — Filters

**Epic:** Order Management

**Confluence:** [Order Management → Story 24.2](https://unifiedinfotech.atlassian.net/wiki/spaces/SBE/pages/3859742741/SBE+-+Admin+Panel#Order-Management) _(page section; Confluence has no per-requirement anchor)_

**Build Status:** 9 requirement(s) to implement here (filter Build Focus = 'Implement here'). 0 covered elsewhere, 0 out of scope.

## Summary

Story 24.2 adds three backend-enforced, combinable filters (date range, Show City, Sales Channel) to the net-new Order Management list, plus an empty state and date validation (End not before Start). There is no order-management module yet, but admin-backend-api already ships a near-identical pattern in the admin/ppl-order list (Order-table query with created_at gte/lte date-range filtering, from<=to validation, pagination, search, status filter, empty result) — so date-range, validation, combinability, empty-state and backend-enforcement are firmly Deliverable by reusing that pattern. The Show City and Sales Channel filters are Partial: neither is a stored Order column. Show City must be derived via OrderItem.show_product_id -> ShowProduct.show_id -> Shows.city_id -> City, which is written only on cart-originated product orders (admin checkout.service + exhibitor cart.service via materializeCartToOrder); subscription/ppl_addon orders and the exhibitor payments.service product/ppl_addon path carry no show_product_id. Sales Channel must be derived from cart.created_by_type (admin = Admin/Sales, exhibitor = Self-Service), available only for orders with a cart_id; subscription/ppl orders and the exhibitor payments.service path have null cart_id and no channel discriminator. Overall feasible now with derivations and a documented coverage gap for non-cart/non-show orders.

## Open Questions

- Sales Channel: the spec defines exactly two values (Admin Role/Sales Order, Self-Service Portal Order). There is no order-level channel column, and orders with null cart_id (subscription/ppl_addon orders, and the exhibitor payments.service product/ppl_addon path) carry no cart.created_by_type discriminator. Should such orders be excluded from the channel filter, or assigned a default channel by origin (e.g. exhibitor-backend orders -> Self-Service)?
- Show City: orders with no show linkage (subscription/ppl_addon orders, and product orders created by exhibitor payments.service which omit show_product_id) cannot be matched to a show city. When a Show City filter is applied, should these orders be excluded entirely (no city match)?

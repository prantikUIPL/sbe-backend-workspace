# SBE-24.1 Implementation Plan — Admin Order Management List (v2 — Draft)

**Scope:** 1 session. One net-new admin listing endpoint `GET /api/v1/orders` in `admin-backend-api`, serving the consolidated Order Management cluster: the list (24.1), filters (24.2), search (24.3), sorting (24.4). Follows `booth-agreements` for module/file/serialization conventions and `ppl-order` for the filter/search/sort/date apparatus.

**Source:** Story feasibility docs `order_management/24.{1,2,3,4}-*`, reconciled by the Phase-1 scope walk (2026-07-02).

**Canon reconciliation:** Story 24.1 lists "booth build" as a type, but there is no `booth_build` producer or `OrderType` value — **24.1-b stays Blocked and is excluded** (tracked in 24.1 Open Questions; SBE-1304/1399/1088). "Total/Balance Due from the centralized billing service" resolved (24.1-n) to authoritative `Order.total`/`paid_amount` — no billing service is built.

**Convention deviations from the primary (booth-agreements):** (a) **Response is wrapped** in the success envelope via `successApiResponse(...)` → `ApiResponseDto<OrderListResponseDto>` (`{ success, message, data: { data, meta }, timestamp }`), NOT the bare `{ data, meta }` that the existing admin *list* endpoints (ppl-order list, booth-agreements) still return. Rationale: `successApiResponse` was just added to `dev` specifically to "mirror the exhibitor API so shared client code sees an identical shape" (util docstring), `ppl-order` already adopted it for its newer endpoints, and the exhibitor sibling SBE-13.2 list is wrapped — this net-new endpoint has no legacy shape to preserve. (b) Swagger success body documents the wrapped shape via `ApiResponseDto` composed with `OrderListResponseDto` (`allOf` + `@ApiExtraModels`), not booth's inline `schema.example` — richer typed model. (c) `totalPages = 0` on empty results (booth uses `Math.max(1,…)`) — matches the SBE-13.2 code-review decision and `ppl-order`. (d) The filter/search/sort/date-range apparatus is taken wholesale from `ppl-order`, as booth-agreements has none.

## Gate Contract

- `GET /api/v1/orders` returns HTTP 200 with the wrapped envelope `{ success, message, data: { data: [...], meta }, timestamp }` (via `successApiResponse`) for a caller holding `orders.list`.
- List scoped to **`order_type = product`** only; `deleted_at: null`. A `subscription`/`ppl_addon` order never appears.
- Caller without `orders.list` (and not `Developer`) → **403**; unauthenticated → **401**.
- Each row: `id`, `order_number`, `customer_name`, `company_name`, `item_total`, `balance_due`, `shows[]` (deduped; `[]` when no show linkage), `sales_person` (nullable), `sales_channel` (`admin_sales` | `self_service`), `order_date`, plus supplementary `status`, `currency`.
- **Filters (24.2):** `from_date`/`to_date` (End ≥ Start else 400), `show_city_id`, `sales_channel` — combinable (AND), backend-enforced; empty match → `data: []`.
- **Search (24.3):** `search`, whitespace-tokenized, each token `contains`+insensitive across `order_number` / `company.name` / `billing_first_name` / `billing_last_name` / `billing_company_name`; tokens AND-ed (B-token).
- **Sort (24.4):** `sort_by` ∈ {`order_number`,`created_at`} (default `created_at`), `sort_order` ∈ {`asc`,`desc`} (default `desc`), full-dataset before pagination.
- `meta` = `{ page, limit, total, totalPages, hasNext, hasPrev }`; `totalPages = 0` when `total = 0`.
- Pure helpers (`deriveBalanceDue`, `deriveSalesChannel`, `buildOrderShows`, `buildCustomerName`, `buildSalesPersonName`, `buildSearchAnd`) + `buildCreatedAtRange` unit-tested.
- `orders.list` seeded in `permission.seeder.ts` and granted to Admin + Super Admin in `role.seeder.ts`.
- All quality gates green (lint, typecheck, jest).

## Context

**Why:** There is no admin management surface for **product** orders (booth / add-on / sponsorship commerce). The only Order list, `ppl-order`, hard-filters to `subscription`/`ppl_addon` and excludes `product` (`ppl-order.service.ts:280`, asserted by its spec). Without this endpoint, 24.5 (Quick Actions) and 24.6 (Order Details) have no list, and admins cannot view/filter/search/sort product orders.

**What:** A new `OrdersModule` in `admin-backend-api` exposing `GET /api/v1/orders`, backed by `OrdersService.listOrders(query)`. Returns a paginated `order_type = product` list over the single `Order` table with read-time derivations: `balance_due` (`total − paid_amount`), `sales_channel` (`cart.created_by_type`), deduped `shows[]` (`orderItems → showProduct → show`), `customer_name` (billing snapshot), `sales_person` name (`salesPerson`). Filters/search/sort are query params. Row `id` is the handle for 24.5/24.6.

## Design Decisions

### D1: New `orders` module — booth-agreements layout, ppl-order apparatus
`admin-backend-api/src/admin/orders/` (`OrdersModule → OrdersController + OrdersService`), registered via `admin.module.ts`. Read-only → injects only `PrismaService`; module `imports: [CommonModule]` (no `AdminAuditModule`, no `exports` — nothing consumes the service yet). **File layout, `dto/index.ts` barrel, `select`-const row shaping, and pagination scaffolding follow `booth-agreements`.** The `OrdersService.listOrders` returns bare `{ data, meta }`; the **controller** wraps it in the success envelope via `successApiResponse(...)` (D2, v2). The **date-range, multi-field/nested search, and sort** apparatus is lifted from `ppl-order` (booth has none).

### D2: Admin host conventions (v2 — wrapped envelope)
- **Wrapped response** (v2, per the `dev` drift). `OrdersService.listOrders` returns bare `{ data, meta }`; the **controller** wraps it: `return successApiResponse('Orders retrieved successfully', result)` → `ApiResponseDto<OrderListResponseDto>` = `{ success, message, data: { data, meta }, timestamp }`. Import `successApiResponse` from `common/utils/api-response.util.ts` and `ApiResponseDto` from `common/dto/api-response.dto.ts` (both added to `dev`; `ppl-order` already uses this pair for its newer endpoints). Controller method return type is `Promise<ApiResponseDto<OrderListResponseDto>>`.
- **Auth is global** (`APP_GUARD`: JwtAuthGuard + RolesGuard). Controller adds only `@Permissions('orders.list')` + `@ApiBearerAuth('JWT-auth')` + **class-level `@ApiAuthResponses()`** (booth style — once, not per-method).
- **Swagger success body** documents the wrapped shape: `@ApiExtraModels(ApiResponseDto, OrderListResponseDto)` + `@ApiResponse({ status: 200, schema: { allOf: [ { $ref: getSchemaPath(ApiResponseDto) }, { properties: { data: { $ref: getSchemaPath(OrderListResponseDto) } } } ] } })`. (ppl-order's wrapped download endpoint uses the simpler `type: ApiResponseDto`, which loses row typing — the `allOf` composition keeps `OrderListResponseDto` accurate. Deliberate deviation from booth's inline `schema.example`.)
- Errors: `@ApiBadRequestErrorResponse(...)` per method.
- Query DTO extends shared `PaginationDto`; meta typed via `PaginationMetaDto`. Defaults (`page=1`,`limit=10`) applied in-service (`?? 1`/`?? 10`).
- Op-description: `[ '**[SBE-24.1 — Order Management List]**', … ].join('\n')`.

### D3: Listing scope = `order_type = product` (not `AND cart_id IS NOT NULL`)
`where.order_type = OrderType.product`, `where.deleted_at = null`. Per the locked decision: product orders always carry `cart_id` by design, but scoping on `order_type = product` alone is the **defensive** predicate — it won't hide a real order if `cart_id` ever goes null (FK is `ON DELETE SET NULL`). Inverts the single `ppl-order` line.

### D4: `select`-const + `GetPayload` row type (booth pattern)
```
const ORDER_LIST_SELECT = {
  id: true, order_number: true, total: true, paid_amount: true,
  currency: true, status: true, created_at: true,
  billing_first_name: true, billing_last_name: true, billing_company_name: true,
  company:     { select: { id: true, name: true } },
  salesPerson: { select: { id: true, first_name: true, last_name: true } },
  cart:        { select: { created_by_type: true } },
  orderItems:  { select: { showProduct: { select: {
                   show: { select: { id: true, title: true, date: true,
                             city: { select: { id: true, name: true } } } } } } } },
} satisfies Prisma.OrderSelect;
export type OrderListRow = Prisma.OrderGetPayload<{ select: typeof ORDER_LIST_SELECT }>;
```
Nested `where` filters (city, sales_channel, search on company.name) are **independent of this select** — `ppl-order` filters `company.is.name` while only selecting `company.{id,name}`.

### D5: Read-time derivations (pure helpers) + a `toListItem` serializer
Unlike booth (returns rows raw), orders carry `Decimal` money and derived fields, so add a `toListItem(row)` that calls pure, reusable helpers in `orders.helpers.ts` (24.6 will reuse them):
- **`deriveBalanceDue(total, paidAmount) → string`** — `(total − paidAmount)`, floored at 0, `.toFixed(2)`.
- **`deriveSalesChannel(createdByType) → 'admin_sales' | 'self_service'`** — `exhibitor → self_service`, else `admin_sales` (null cart → `admin_sales` fallback, documented).
- **`buildOrderShows(orderItems) → OrderShow[]`** — distinct `showProduct.show` deduped+sorted by id; `[]` when none (24.1-h).
- **`buildCustomerName(first, last) → string`** — trimmed `"first last"`.
- **`buildSalesPersonName(salesPerson) → string | null`** — trimmed name or `null`.
- **`buildSearchAnd(search) → Prisma.OrderWhereInput[]`** — whitespace-tokenize; each token → `OR` across the 5 text fields (`contains`+`insensitive`); array for `where.AND` (B-token).
`toListItem` also emits `item_total = total.toFixed(2)`, `order_date = created_at`, `company_name = billing_company_name ?? company.name`.

### D6: Item Total = `Order.total`
`item_total = Order.total.toFixed(2)`; `balance_due = deriveBalanceDue(total, paid_amount)` — authoritative Order columns (24.1-n). Fixed-2 strings (ppl-order convention), no recompute. If the client later means pre-tax `subtotal`, it's a one-field swap.

### D7: Filter where-builder (24.2, combinable AND)
On the D3 base:
- date → `buildCreatedAtRange(from_date, to_date)` (ported verbatim from ppl-order: ISO parse, `gte`/`lte`, from>to → 400) → `where.created_at`.
- `show_city_id` → `where.orderItems = { some: { showProduct: { is: { show: { is: { city_id } } } } } }`.
- `sales_channel` → `where.cart = { is: { created_by_type: self_service ? exhibitor : admin } }`.
- `search` → `where.AND = buildSearchAnd(search)`.
All AND-compose (24.2-f); empty match → `data: []` (24.2-g); all server-side (24.2-h).

### D8: Sort contract (24.4)
`OrderSortField` {`order_number`,`created_at`} default `created_at`; `OrderSortOrder` {`asc`,`desc`} default `desc`. `orderBy: { [sort_by]: sort_order }` before `skip`/`take` (full-dataset, 24.4-c). Whitelisted enums (injection-safe). snake_case params.

### D9: RBAC via a new `orders.list` permission
No `orders.*`/`order.*` key exists. Add `{ module: 'orders', method: 'GET', url: '/orders', permission_key: 'orders.list', is_public: false, is_visible: true }` to `permission.seeder.ts` (new "Admin – Order Management" block ~line 1219); add `'orders.list'` to `adminPermissions` in `role.seeder.ts` (Admin + Super Admin; `Developer` bypasses). No schema migration — seed data only; re-run the seeder.

### D10: `totalPages = 0` on empty (deviation from booth primary)
`totalPages = Math.ceil(total / limit) || 0` (ppl-order rule) → **0 when `total = 0`**, per the SBE-13.2 post-review decision. Explicitly overrides booth's `Math.max(1, …)`; the empty-case service test asserts `totalPages: 0`.

### D11: Concurrency hazard analysis
Read-only (`findMany`/`count`), no writes/locks/emitters, no auto-triggered actions, no concurrency-constant changes → **no mitigations required**.

## Public API Contract

### Caller Contract
`GET /api/v1/orders?page=&limit=&search=&from_date=&to_date=&show_city_id=&sales_channel=&sort_by=&sort_order=` — admin JWT + `orders.list`. Lists **product** orders across all companies (admin scope). Unknown params → 400 (global `ValidationPipe whitelist`).

### Response shape (v2 — wrapped)
```jsonc
{
  "success": true,
  "message": "Orders retrieved successfully",
  "data": {
    "data": [{
      "id": 123,
      "order_number": "ORD-2026-00123",
      "customer_name": "Jane Smith",
      "company_name": "Acme Corp",
      "item_total": "1500.00",
      "balance_due": "500.00",
      "shows": [{ "id": 7, "title": "Small Business Expo NYC", "city": "New York", "date": "2026-09-10T00:00:00.000Z" }],
      "sales_person": "Alex Rep",
      "sales_channel": "self_service",
      "order_date": "2026-06-01T10:00:00.000Z",
      "status": "partially_paid",
      "currency": "usd"
    }],
    "meta": { "page": 1, "limit": 10, "total": 1, "totalPages": 1, "hasNext": false, "hasPrev": false }
  },
  "timestamp": "2026-07-02T12:00:00.000Z"

```

### Consumer Responsibilities
- `shows: []` = no linked show (render dash).
- Row `id` is the handle for 24.6 (details) / 24.5 (quick actions) — targets not provided here.
- Money fields are fixed-2 strings; do not recompute.

## Security Considerations
- **RBAC (24.1):** requires `orders.list` via global `RolesGuard`; missing → 403, unauth → 401 (class-level `@ApiAuthResponses`). Granted to Admin + Super Admin (+ Developer bypass) in `role.seeder.ts`.
- **Input hardening:** pagination bounded by `PaginationDto` (`@Min/@Max(100)`); `search` trimmed + `@MaxLength(100)`; dates `@IsISO8601`; `show_city_id` coerced int; enums whitelisted; global `forbidNonWhitelisted` rejects unknown params.
- **Injection-safe sort:** `sort_by` is a closed enum.
- **PII:** rows carry only summary columns — no billing address, card, or Stripe ids.

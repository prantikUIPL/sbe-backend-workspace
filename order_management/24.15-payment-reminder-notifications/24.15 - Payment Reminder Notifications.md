## Summary

An automatic, backend-enforced reminder system nudging customers about upcoming and overdue **manual** payments (check / wire / ACH — 24.8's `payment_type != credit_card`), repeating until the installment is settled, stopping automatically the moment 24.8's mark-as-Paid PATCH (→ `succeeded`) or 24.6's void endpoint (→ `canceled`) flips the row. The 2026-07-03 gate locked: **independent worker cron** (D4 — not waiting for the unstarted E&S dynamic-scheduling system, which claims payment-due reminders; hand-off note filed in the *scheduling* register), cloned from the proven payment-charge registrar/task pair; **email-only** (D1 — SendGrid is the only real dispatch path; SMS is an enum value with no client anywhere); **settings-driven cadence** (D2 — `payment_reminder_upcoming_days` / `payment_reminder_overdue_interval_days` / `payment_reminder_hour_utc`, proposed defaults **7 / 7 / 14:00 UTC pending BA ratification** — the spec contains NO cadence, window, or hour anywhere; per gate directive these values and their provenance MUST be documented in code); **manual-rows-only predicate** (24.15-b option a — retry-exhausted card rows excluded, already covered by the per-failure email; register note); slugs **`payment_reminder_upcoming` / `payment_reminder_overdue`, seeded by 24.15 itself** per the template-ownership rule (flows own their templates; the E&S epic only manages them). Stop/continuation needs **no new state**: the cron re-selects unpaid rows each tick and checks `PT.status` at send time — termination auto-materializes because 24.8 builds first. Recipient = order `billing_email` (24.11 chain precedent); CC of `Order.additional_emails` pending the consolidated OQ-2 BA row shared with 24.6.

## Decisions (locked at the 2026-07-03 gate)

- **D1 — Channel: EMAIL-ONLY.** Worker mailer is SendGrid-only; `NotificationChannel.SMS` has no send client in any repo; story text never asks for SMS; enum accommodates SMS later without migration.
- **D2 — Cadence: settings-driven mechanism locked; values proposed pending BA.** `ppl-settings` family mirroring `payment_charge_hour_utc`: `payment_reminder_upcoming_days` (default **7** — days before due, single send), `payment_reminder_overdue_interval_days` (default **7** — repeat interval after due, dedupe per interval via notification_logs), `payment_reminder_hour_utc` (default **14** — business-hours send, unlike the deliberately off-peak 02:00 charge hour). **The spec specifies NOTHING here** (re-confirmed on the live row): per gate directive, the settings seeder AND the cron service carry doc-comments stating these defaults were set at the 2026-07-03 gate, not derived from Confluence; what each knob does; the dedupe rule; and that BA ratification is pending (OQ-1). BA's answer changes a seeded default, not code. Also resolves the "System, payment reminders" nav-clause tension cheaply: the settings family IS the future config surface if ever wanted; 24.15 ships UI-less.
- **D3 — Interim predicate: RESOLVED-BY-SEQUENCING.** 24.15 builds after 24.8 in the release, so the manual-rows predicate (`payment_type != credit_card`) ships from day one — no dark launch, no courtesy-email interim, no coupling risk. (Old options a/b retired.)
- **D4 — Independent of the E&S dynamic-scheduling system.** That design (docs complete under `email_and_sms_docs/email_sms_scheduling_plan_and_supporting_docs/`, implementation NOT started) claims payment-due reminders (ANCHOR_RELATIVE). 24.15 ships as a deliberately minimal independent cron now; a hand-off note goes in the **scheduling** known-issues register (per the register-separation rule — NOT the main E&S register) so the future build absorbs/supersedes this cron knowingly.
- **D5 — Slugs: `payment_reminder_upcoming` / `payment_reminder_overdue`, seeded by 24.15.** Ownership rule (locked at 24.11): each flow's story seeds its own slugs/templates. SBE-671 is MERGED — the old unmerged-branch collision concern is retired; dev's 29 live trigger slugs have zero reminder entries (verified). Seeder discipline: append-don't-reorder; the fail-loud META guard catches collisions at the gate.
- **24.15-b sub-decision — manual rows only (option a).** Retry-exhausted card rows (`failed AND retry_count=3` — permanently stuck) are genuinely customer-actionable but EXCLUDED: the story's framing is manual payments, and stuck card rows already receive the per-failure email with update-card CTA. Register note lets BA extend later.
- **Recipient (carried from 24.6 OQ-2):** to = order `billing_email` (24.11 chain precedent: billing_email → company contact → skip+log); CC of `additional_emails` pending the consolidated OQ-2 BA row — if ratified, a one-line CC-append in this cron's dispatch.

## Build Consolidation Notes (avoid redundant work)

- **Clone, don't invent:** registrar/task/service triple cloned from `payment-charge` (same settings-hour pattern, same overlap guard); dispatch via the **worker** mailer (`sendFromTemplate` :88) — NOT admin's mailer (that serves 24.6-r/24.11/24.5 action emails; both write notification_logs — keep notification_type consistent).

Reminder selection must NOT assume Stripe columns present (24.8-D3 made stripe_pm/cus nullable for manual rows).
- **No new state anywhere:** triggers, continuation, and stop are all reads of PT (`status`, `due_date`, `payment_type`) + notification_logs dedupe. The story adds a cron, two templates, three settings, one dev trigger — nothing else.
- **Templates:** seeded by 24.15 per the ownership rule; append-don't-reorder in both seeders (merge hotspots; fail-loud META guard is the backstop).

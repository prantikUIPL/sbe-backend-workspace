## Design Decisions

### D1: Net-new `internal_notes` column — `Order.notes` is untouchable

`Order.notes` is the exhibitor checkout **idempotency-key store**: both exhibitor checkout paths write `idempotency_key:<key>` at order creation (`cart.service.ts:1277`, `payments.service.ts:213`) and substring-match it for 24h duplicate-order/charge replay protection (`:1541`, `:465`). Admin-editing it would break replay protection and render machine tags in the UI.

Implication: 24.14-a stores to a new `internal_notes` column. The notes endpoints must **never select, return, or write `Order.notes`** — enforce via a frozen SELECT constant that excludes it, and a unit test asserting the constant's keys.

### D2: Four nullable columns; Cart definitions copied verbatim; blank at checkout

`internal_notes @db.Text`, `payment_memo @db.Text` (Text over VarChar — human decision 2026-07-02), `invoice_note @db.VarChar(255)` and `additional_terms @db.Text` copied exactly from Cart (schema:2853-2854) so the same field has one shape across the domain. One combined `invoice_note` (not separate `po_number`) per the Cart precedent. Nullable **by design** (admin-entered-later fields with no backfill source; Cart originals are nullable — agreed deviation from the usual NOT-NULL+backfill preference). No checkout seeding (resolved 2026-07-02): zero changes to any order-creation path.

**v2 change (review finding):** the four columns must ALSO be added to the other four repos' `schema.prisma` Order models (schema-only, no code) — those repos apply schema via `prisma db push`, which would DROP columns absent from their schema. Carrying the columns ≠ exposing them: no sibling SELECT touches them. Cart's own `internal_notes` (admin schema:2852, "CS-only") is added precedent for the column name/type.

### D3: Single sub-resource endpoint pair; PATCH = partial update, null clears

`GET`/`PATCH /api/v1/orders/:id/notes` on the shipped `OrdersModule` — requirements a–d are fields in one DTO, not separate endpoints. PATCH semantics:

- All four DTO fields optional. **Omitted (undefined) → untouched; explicit `null` → cleared**; strings are trimmed, and a trimmed-empty string normalizes to `null`.
- Empty payload (no recognized field present) → 400.
- Validation: `@IsString` + `@MaxLength(255)` on `invoice_note` (DB cap); `@MaxLength(5000)` on the three Text fields (mirrors the account-history-notes DTO cap; DB is unbounded).
- Response: module convention `successApiResponse(...)` envelope (orders-module style, not account-history-notes' raw return).

### D4: Per-field audit entries via `recordMany`, no-op-aware

Inside one `$transaction`: read the current row (404 discipline) → `order.update` with only the provided fields → build one `AdminAuditRecordInput` **per field whose value actually changed** (`previousValue` = old raw value, `newValue` = new raw value, note e.g. `'Payment Memo updated.'`, `performedBy` = JWT admin id) → `adminAudit.recordMany(entries, tx)`. A field submitted with its current value produces **no** audit entry; a payload producing zero changed fields still returns 200 (idempotent PATCH) with no audit rows. Audit permanence/display need no work (append-only table; existing `GET /logs/admin-audit?entity_type=order&entity_id=:id`).

### D5: Product-order scoping + ownership-404 discipline

`findFirst({ where: { id, deleted_at: null, order_type: product }, select: ORDER_NOTES_SELECT })` → `NotFoundException('Order not found')` otherwise — matching the shipped list's product-only predicate (`orders.service.ts:104-106`) so the notes surface covers exactly the orders the OM UI can reach. **Approved 2026-07-02: product-only stands** (matches the OM surface; revisit only if the team extends OM to other order types).

### D6: RBAC + Swagger documentation of permission keys

Seed `orders.notes.read` (GET `/orders/:id/notes`) and `orders.notes.update` (PATCH) in the `permission.seeder.ts` 'orders' block; append both keys to every `role.seeder.ts` array that grants `orders.list` (grep-driven — currently one block, `:178`). Follow the shipped orders precedent of **no** permission-group entry (SBE-1125 added none for `orders.list`). Each route's `@ApiOperation` description **must state its permission key** (human requirement 2026-07-02) — e.g. `'Requires permission: orders.notes.update.'` line, alongside `@ApiBearerAuth`.

### D7: Dedicated `OrderNotesService` in the orders module

New `order-notes.service.ts` beside `orders.service.ts` (mirrors the single-responsibility account-history-notes service; keeps the list service untouched). `OrdersModule` adds `AdminAuditModule` to imports (it exports `AdminAuditService`) and registers the new provider. No new Swagger wiring — `OrdersModule` is already in the `createDocument` include list with the 'Admin - Order Management' tag.

## Public API Contract

### Routes

| # | Method | Path | Status | Permission | Notes |
|---|--------|------|--------|------------|-------|
| 1 | GET | `/api/v1/orders/:id/notes` | 200 | `orders.notes.read` | Notes bundle for the admin Notes tab |
| 2 | PATCH | `/api/v1/orders/:id/notes` | 200 | `orders.notes.update` | Partial update; per-changed-field audit |

**GET 200 envelope `data`:** `{ id, order_number, internal_notes, payment_memo, invoice_note, additional_terms }` (all four note fields `string | null`).

**PATCH request body:** `{ internal_notes?, payment_memo?, invoice_note?, additional_terms? }` — each `string | null`; ≥1 field required. **PATCH 200 envelope `data`:** same shape as GET (post-update).

**Error mapping:** invalid `:id` / body → 400 (`ParseIntPipe` + class-validator); missing/invalid JWT → 401; missing permission → 403; unknown / soft-deleted / non-product order → 404 `'Order not found'`.

**Consumer responsibilities (24.6):** read the four columns in its own detail SELECT; route edits through this PATCH (canonical write path); never touch `Order.notes` (D1).

### Step 1: Migration — `prisma/migrations/20260702120000_add_order_admin_note_columns/migration.sql`

Add to the `Order` model (after `notes`, keeping the block's alignment): `internal_notes String? @map("internal_notes") @db.Text /// Admin-only internal notes (24.14; NOT Order.notes — that is the checkout idempotency tag)`, `payment_memo String? @map("payment_memo") @db.Text /// Admin payment memo for accounting reference`, `invoice_note String? @map("invoice_note") @db.VarChar(255) /// Invoice Note / PO Number`, `additional_terms String? @map("additional_terms") @db.Text /// Additional Terms & Conditions`.

SQL: single `ALTER TABLE "orders" ADD COLUMN ...` with the four columns. No indexes, no backfill, no enum changes. Verify with `prisma migrate dev` against the local DB, then `prisma generate`.

### Step 2: DTOs — `src/admin/orders/dto/order-notes.dto.ts`

`UpdateOrderNotesDto`: four fields, each `@IsOptional() @IsString() @MaxLength(cap)` (255 for `invoice_note`, 5000 for the rest), `@ApiPropertyOptional({ nullable: true })`. Nulls pass through (class-validator skips null under `@IsOptional`). `OrderNotesResponseDto`: `id`, `order_number`, four nullable string fields, `@ApiProperty` annotated for the envelope schema (`ApiExtraModels` + `getSchemaPath` per the list route's pattern). Export from `dto/index.ts`.

### Step 3: `OrderNotesService` — `src/admin/orders/order-notes.service.ts`

- `ORDER_NOTES_SELECT = { id, order_number, internal_notes, payment_memo, invoice_note, additional_terms } as const satisfies Prisma.OrderSelect` — frozen constant; **excludes `notes`** (D1).
- `AUDIT_LABELS: Record<field, string>` — `'Internal Notes' | 'Payment Memo' | 'Invoice Note / PO Number' | 'Additional Terms & Conditions'`.
- `getNotes(orderId)` — `findFirst` per D5 → 404 or DTO.
- `updateNotes(orderId, dto, performedBy)` — normalize (trim; `'' → null`; drop undefined) → 400 `BadRequestException` if no field present → `$transaction`: re-fetch row (D5 select, 404), `tx.order.update({ data: changedFieldsOnly })`, build per-changed-field audit inputs (`entityType: AdminAuditEntityType.order`, previous/new raw values, `note: '<label> updated.'`), `adminAudit.recordMany(inputs, tx)` → return post-update DTO. Zero-changed-fields short-circuits to a plain read (200, no write, no audit).

### Step 4: Controller + module wiring

Two handlers on `OrdersController` under `@Param('id', ParseIntPipe)`, mirroring the account-history-notes controller shape but with the orders module's envelope + Swagger style: `@Permissions('orders.notes.read'/'orders.notes.update')`, `@ApiOperation` description **including the permission key line** (D6) and the `[SBE-24.14 — Internal Notes & Administrative Information]` story header line (list-route convention), `@ApiResponse` with `ApiResponseDto` + `OrderNotesResponseDto` schema composition, `@ApiBadRequestErrorResponse`/`@ApiNotFoundErrorResponse` decorators. `orders.module.ts`: add `AdminAuditModule` to imports, `OrderNotesService` to providers.

## Security Considerations

- Admin-only surface: global JWT + roles + `@Permissions` stack; keys seeded to the same roles as `orders.list` and stated in Swagger (D6).
- Customer isolation (24.14-f): new columns exist only in admin's schema; exhibitor's frozen SELECT constants can't leak them.
- `Order.notes` reservation (D1) prevents an admin-facing write from corrupting checkout idempotency (duplicate-charge protection).
- Audit rows capture raw previous/new values — internal admin data in an admin-only log; no PII beyond what admins typed.

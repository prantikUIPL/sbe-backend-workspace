## Summary

Story 24.14 asks for an admin-only "Notes" surface on the Order Details page where admins add/edit Internal Notes, a Payment Memo, Invoice Notes / PO Number, and Additional Terms & Conditions on an order, with every note addition/modification captured in a permanent, non-editable, backend-enforced audit trail. Re-verification against dev (2026-07-02) confirms every supporting primitive is built: the shipped Order Management module (`src/admin/orders/`, SBE-1125) hosts the new routes; the append-only AdminAuditLog + transaction-aware AdminAuditService.record()/recordMany() pattern is fully implemented with `order` already in the entity enum; and account-history-notes is a working identical template (transactional note-column update + audit write). **One material correction from the scope walkthrough:** `Order.notes` **cannot** be reused for Internal Notes — it is the exhibitor checkout **idempotency-key store** (both exhibitor checkout paths write `idempotency_key:<key>` into it at order creation and substring-match it for 24h duplicate-order/charge protection), so 24.14-a gets a **net-new `internal_notes` column** and `Order.notes` stays untouched. The single migration therefore adds **four nullable columns** to `orders`: `internal_notes Text`, `payment_memo Text`, `invoice_note VarChar(255)` (copied verbatim from Cart), `additional_terms Text` (copied verbatim from Cart). The API surface is a **single endpoint pair** — GET + PATCH `/api/v1/orders/:id/notes` — covering all four fields (per-field audit entries via recordMany inside the PATCH transaction). The cart-seeding open question was **resolved 2026-07-02: leave the new columns blank at checkout** (admin-entered on the Notes tab; no write-path changes in any repo). Overall verdict: fully Deliverable now, zero external dependencies.

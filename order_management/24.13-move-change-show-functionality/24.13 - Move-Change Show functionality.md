# Story 24.13 — Move/Change Show functionality

**Epic:** Order Management

**Confluence:** [Order Management → Story 24.13](https://unifiedinfotech.atlassian.net/wiki/spaces/SBE/pages/3859742741/SBE+-+Admin+Panel#Order-Management) _(page section; Confluence has no per-requirement anchor)_

**Build Status:** 9 requirement(s) to implement here (filter Build Focus = 'Implement here'). 0 covered elsewhere, 1 out of scope.

## Summary

The story asks for an Admin "Move Show" action: a direct, no-edit transfer of an existing order's entire package (all booth sizes + all included products) from one city/show to another, validating destination inventory first and failing with a clear message when matching inventory is unavailable, enforced on the backend. The data primitives are real and built: Order/OrderItem (with show_product_id), per-show ShowProduct offerings (admin schema.prisma:2640, @@unique([show_id, product_id]) :2660), and a genuine inventory engine (InventoryService.getAvailability / assertAvailable / commitForCart with a FOR UPDATE row lock). Re-verification overturned the prior analysis's central gap: the inventory RELEASE write-path IS implemented — external-api-service webhook.service.ts releases committed reservations (status->released, keyed by order_id) on first-installment payment failure (:2760) and payment_intent.canceled (:2858), and the same webhook transitions Order status to failed/canceled. There is still NO order-management admin module/controller (no admin order read/update/cancel/move endpoint) and no admin-initiated cancel, so the Move Show endpoint + Order Management surface are net-new, but they sit entirely on working primitives. The transfer, inventory validation, failure message, and atomic release+re-commit are all Deliverable on built code; the only genuine gap is included-product availability, which has no inventory backing because default-included lines are deliberately excluded from inventory commit (Partial).

## Open Questions

- On a successful move, are the order's financials re-priced to the destination show's ShowProduct prices (which differ per show via sales_price/actual_price) or kept frozen at the original order totals? The spec mandates 'no modification' but is silent on price impact.
- Does 'validate applicable included products' mean only an existence check (the destination show offers each included product_id) or per-show stock validation? Default-included lines carry no tracked inventory (excluded from commit; decreasing/removing never changes totals), so there is no inventory to validate against beyond existence — confirms the intended semantics for 24.13-f.

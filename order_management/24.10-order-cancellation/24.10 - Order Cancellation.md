## Summary

Admin "Cancel Order" on the Order Details page: after explicit confirmation, cancel the whole order with full or capped-partial refund — set the order to Cancelled, deactivate unpaid/pending installments, retain paid installments as history, process Stripe/manual refunds, enforce + log on the backend. Re-verified 2026-07-03 on origin/dev (admin `0ac2769`, exhibitor `8c70e36`, external `8b884ea`): the cancellation mechanics are all built primitives (status enums + proven `updateMany` write-paths, webhook-maintained `paid_amount` cap source, tx-aware audit service with `order` entity precedent). The orders module is shipped, so the cancel endpoint is an **additive route**.

**All refund work is delegated to 24.9 (D1, locked 2026-07-03):** 24.9 builds first in the same release and owns the Refund ledger + `refund_failed` migration, the refunds service/endpoint (per-installment primitive + order-level composition), the Stripe wrapper, and the amount-aware `charge.refunded` handler. 24.10's cancel flow **calls 24.9's order-level composition** (newest-first legs) **after its DB transaction commits** — Stripe HTTP cannot sit inside a `$transaction`, so the order is already `canceled` when the refund legs run; a refund failure leaves the order canceled with a precise error, no intent-only interim, no contract re-open. Former Partial/Blocked rows (d, f, g, h) upgraded to Deliverable-sequenced accordingly; 24.10-g's producer corrected to 24.8 → 24.7 (D2).

**Scope addition (24.10-k):** release committed inventory reservations in the cancel transaction. Absorbed from OOS 24.12 as a data-integrity invariant, not a feature: both existing order-death paths release reservations (webhook payment-failure paths — the only release writers anywhere), so a cancel that skips release strands units forever (availability sums committed rows with no expiry). Housed as reusable `InventoryService.releaseForOrder(orderId, tx)` for 24.13 next sprint.

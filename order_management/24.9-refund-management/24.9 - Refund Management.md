## Summary

Story 24.9 asks for an admin Refund capability under Order Management that issues either a Manual (offline) refund or a Stripe refund against an order's payment plan, capped at the amount paid, with a mandatory reason, reflecting Stripe success/failure states. Re-verification against origin/dev (2026-07-03: admin `0ac2769`, external `8b884ea`) confirms every read-side primitive is BUILT and populated: `Order.paid_amount` is a webhook-maintained running sum, `PaymentTransaction.amount`/`stripe_charge_id` identify per-installment charges, the Stripe SDK is wired in external-api-service, `refunded` exists in both status enums, and the inbound `charge.refunded` webhook already flips a PT + voids its invoice. The orders module itself is now SHIPPED on admin dev (24.1‚Äì24.4 + 24.14), so 24.9 is additive routes, a net-new Refund ledger model + `refund_failed` enum value, and the net-new outbound `stripe.refunds.create` wrapper ‚Äî all story-owned work on working primitives.

**Design decisions locked 2026-07-03:** (D1) refunds are **per-installment as the primitive** (a Refund row targets one PaymentTransaction; cap = that PT's amount minus its prior refunds; `refund_failed` is added to **PaymentTransactionStatus**), with **order-level refunds as a composition** (fan-out over installments newest-first; `OrderStatus.refunded` becomes a derived state once all settled installments are refunded) ‚Äî this serves both 24.8's per-installment Refund action and 24.10's one-amount cancel UX. (D2) the amount-blind `charge.refunded` webhook handler is made **amount-aware inside this story** (cross-repo change to external-api-service pulled into SBE-1133 scope): full refund ‚Üí flip PT + void invoice as today; partial ‚Üí PT status untouched, the Refund ledger carries the accounting. Verified 2026-07-03: no partial-refund logic exists anywhere, no code path issues refunds today (dashboard-only), so 24.9 is the system's first refund producer. (D3) `Order.paid_amount` **stays gross** (never decremented by refunds ‚Äî consistent with all existing readers); "net paid" derives as `paid_amount ‚àí Œ£ Refund rows` wherever needed.

The single genuine external gap is unchanged: no write-path records offline (check/wire/cash) payments as paid, so the manual-only branch of 24.9-f stays Partial until 24.8/24.7 ship the manual-payment lifecycle. QuickBooks refund sync (24.9-i) stays Out of Scope ‚Äî the QB home is the un-ticketed epic 65 on Admin Panel Page 2, which itself has **no refund/credit-memo story** (flagged for BA).

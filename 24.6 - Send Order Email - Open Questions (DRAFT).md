## 1. What the spec asks (item #7, verbatim intent)

> **Send Order Email** — On the Order Details page, the admin selects an email template from a dropdown, then either saves the order or sends the email.
> Dropdown options: **New Order Confirm/Invoice · Cancelled Order · Invoice/Payment Reminder with link.**

This is a **manual, admin-triggered** email send. The complication is that two of the three options overlap with **already-delivered automated email flows**, and the third ("with link") depends on infrastructure that doesn't exist yet. Those overlaps and gaps are what need client direction before build.

## 3. Current build state (why these questions exist)

Investigated across all five repos + the delivered story docs. Summary:

- **No manual "send order email with a chosen template" endpoint exists.** The only on-demand order emails today are two fixed actions: `POST /orders/:id/resend-confirmation` (always the booth confirmation, invoice **PDF attached**, no template choice, throttled 5 min) and `POST /orders/:id/resend-portal-password`. There is no template dropdown, no order-scoped template-list endpoint, and no `orders.send_email` permission.
- **"New Order Confirm/Invoice"** already **auto-sends at checkout** — so the manual option is a **re-send**, not a new email.
- **"Cancelled Order"** email (`order_canceled`) exists but fires **automatically as a side-effect of the cancel/refund action** (Story 24.11, SBE-1135) — there is no standalone "just send the cancellation email" path. This flow has **no dedupe** — a manual re-send would silently deliver a duplicate.
- **"Invoice/Payment Reminder with link"** is the largest gap: the only reminder templates (`payment_reminder_upcoming` / `payment_reminder_overdue`, Story 24.15, SBE-1139) are **cron-only**, fire **only for split-payment, non-credit-card installments**, and contain **no link or PDF at all**. They do have a coarse email+slug+time-window dedupe, which a manual send could interfere with (one suppressing the other).
- **There is no signed/emailable invoice link anywhere.** The only invoice URL is a raw, **unsigned CloudFront URL** with a **guessable** key (`exports/invoices/{company_id}/{invoice_number}.pdf`); the code is internally contradictory about whether that URL is even publicly openable. Exhibitor download is auth-gated but ultimately returns the same raw URL. Current admin emails sidestep this by **attaching the PDF bytes** rather than linking.

**Net:** item #7 is effectively unbuilt, and building it naïvely would duplicate/conflict with 24.11 and 24.15 and lean on a non-existent secure invoice link. Hence the questions below.

## 4. Client questions — detail

**CQ1 — Purpose & per-option availability of the dropdown.**
Two of the three options duplicate emails the system already sends automatically: "New Order Confirm/Invoice" already auto-sends at checkout, and "Cancelled Order" already auto-sends when an order is cancelled/refunded (24.11). So the manual dropdown is effectively a **re-send** control.
*Please confirm:* (a) that re-send is the intent (not a new/different email); and (b) whether each option should be **enabled only when it applies to the order's state** — e.g. "Cancelled Order" only for cancelled/refunded orders, "Confirmation" not for cancelled orders, "Reminder" only when a balance is due.
*Recommendation:* Yes to both — treat all three as re-sends, and gate each option on the relevant order state so an admin can't send a contradictory email.

**CQ2 — Scope of the manual payment reminder.**
The automated reminder covers **only split-payment, non-credit-card installments**. The manual option's scope is unspecified.
*Please confirm:* which orders qualify for a manual reminder — any order with an outstanding balance? credit-card installment plans too? one-time (non-plan) invoices?
*Recommendation:* Simplest is any product order with an outstanding balance, regardless of payment method.

**CQ3 — The invoice / payment link ("with link").**
"Invoice/Payment Reminder with link" and "New Order Confirm/Invoice" imply an emailable link, but no secure emailable link exists today.
*Please confirm two things:*
1. Is "the link" a **pay-now / payment** link, or an **invoice-PDF** link?
2. How should it be delivered — a **secure, time-limited link** (needs to be built), or **attach the PDF** (works today, matches the current confirmation email)?
*Recommendation:* A secure, time-limited link for anything customer-facing; attach the PDF as an interim if that can't make the release. A **permanent public link is not recommended** — the current invoice URL is guessable and never expires, so anyone with the URL (forwarded or guessed) could view the invoice's billing details with no login.

# Email & SMS Management — Release Notes & Known Issues

## What's included in this release

The Email & SMS Management module delivers a working admin API for managing notification templates. The following is functional now and ready for QA:

- **Template listing** — `GET /api/v1/notification-templates` returns paginated rows (name, type, tag, channel, active status, predefined/custom flag, last-updated, trigger-event label) with full pagination metadata, sorted most-recently-updated first (ties broken by newest id). One endpoint serves both the predefined and the custom listing views via a predefined/custom filter.
- **Search and filters** — trimmed, case-insensitive partial search (up to 254 characters) across template name, trigger-event label, subject, and trigger slug (a whitespace-only search value is treated as no search and returns the unfiltered list); multi-select Type and Channel filters (repeated params or comma-separated) plus Active-status and predefined/custom filters, with OR within a filter and AND across filters and search; invalid values return a clean 400.
- **Template detail** — `GET /api/v1/notification-templates/:id` returns the full record including subject, body with placeholders stored unresolved, sending configuration, and the joined trigger event with its allowed-placeholder list; empty optional fields come back as explicit nulls.
- **Create custom email templates** — `POST /api/v1/notification-templates` (201 + Location header): required trigger validated against the trigger-event catalog, every `{{placeholder}}` validated against that trigger, FROM address restricted to the active whitelisted domains (theshowproducers.com, thesmallbusinessexpo.com), 1–50 TO recipients, optional CC/BCC de-duplicated case-insensitively, HTML body sanitized on write, and the new row appears in the listing immediately.
- **Edit with two-tier rules** — `PUT /api/v1/notification-templates/:id`: predefined templates lock the trigger, FROM address, and TO recipients (subject, body, name, status, from-name, reply-to, CC, BCC are editable); custom templates can additionally edit the trigger, FROM address, and TO recipients — re-pointing the trigger re-validates the stored subject/body placeholders. On both tiers, `channel`, `language`, `schedule_config`, and `follow_up_config` are never editable: the detail endpoint returns them, but they are absent from the edit contract, so including them in a PUT (for example by round-tripping a detail response back into the edit endpoint) is rejected with a field-specific 400 ("property X should not exist"). No-op edits are detected and write nothing.
- **Delete custom templates** — `DELETE /api/v1/notification-templates/:id` removes custom templates (with audit snapshot and cleanup of related send-log rows); deleting a predefined template is rejected with a 400.
- **Dropdown endpoints** — `GET /api/v1/trigger-events` (the 21-event catalog with per-event placeholder lists; read-only, code-controlled) and `GET /api/v1/allowed-from-domains` (active whitelisted FROM domains).
- **Audit trail** — every create writes a snapshot record, every edit writes one record per changed field (down to individual config keys), every delete writes a pre-delete snapshot, all with the acting user and all transactional; history is readable through the central admin audit-log endpoint filtered by entity type and id.
- **Hardened id handling** — non-numeric, out-of-range, hex (`0x10`), exponent (`1e2`), and padded id strings on the template routes return a 400; bad ids never cause a 500.
- **Seeded data** — 18 predefined email templates (17 from the original catalog plus the late-added Cart Proposal Updated), 21 trigger events, and 2 allowed FROM domains. The template seeder is create-only, so re-running it never overwrites admin edits.

## Known issues and limitations

### Deferred to a later phase

1. **Scheduling and follow-up sends are not built.** The Email and SMS Scheduling stories (frequency, days-after-trigger offsets, follow-ups) were deferred by agreement with the business analysts, and no scheduler exists anywhere in this release. QA will see schedule and follow-up fields always returned as null on the detail endpoint, with no way to set them via the API; the database columns are already reserved so no migration is needed when this is built later.

2. **No SMS provider is integrated, so nothing can ever send as a text message.** This is a client dependency deferred to a later phase; the data model supports the SMS channel and the predefined-SMS edit restrictions exist in code, but there is no dispatch path. QA should expect no SMS delivery or delivery-status behavior anywhere in the system.

3. **Dynamic Recipient Resolution is deferred for custom templates, and the matching predefined-template story does not apply.** Recipient fields accept only literal, format-validated email addresses — tokens like `{salesperson}` or `{all customer contacts}` are rejected with a 400, and no resolution engine exists in any service. For predefined templates the FROM/TO are system-controlled and read-only by design, so a resolution engine there would contradict the agreed design; the business analysts have been asked to remove that story rather than defer it.

4. **The recipient picker has no backing source lists.** The stories describe choosing TO/CC/BCC from predefined lists of admin users, exhibitors, and other system emails, but no endpoint serves those lists from this module, and "other relevant system emails" has no known data source at all. Until those source lists are defined and exposed by their owning modules, recipients are free-typed email addresses only.

5. **No in-editor image upload.** Asset hosting belongs to a separate module, so this release stores the provided HTML body only; images work solely as references to already-hosted URLs. QA should not expect any upload capability when testing template bodies.

### Template catalog coverage (client wish-list vs. what is seeded)

6. **Only 4 of the client's approximately 41 requested templates are delivered and live in this release.** The seeded catalog was built from the existing live production email flows, not from the client's wish-list, so the overlap is small. The delivered overlap is:

   | Client template | Delivered as | Note |
   |---|---|---|
   | New User Registration (Welcome Email) | Exhibitor Welcome (Self-Registered) **and** Exhibitor Welcome (Admin-Created) | One client row, split into two variants by registration path |
   | Password Reset Request (Reset Link Email) | Exhibitor Password Reset | Client-facing reset link (the separate admin-panel reset is a system template) |
   | Additional User Added to Company (Invitation Email) | Company User Invitation | Covers both the exhibitor-invite and admin-add paths |
   | New Order Created (PPL Subscription) | PPL Order Confirmation | Only the order-confirmation email; the row's internal sales alert and SMS confirmation are not covered |

7. **Most client-requested templates are not seeded because their owning modules are not integrated with the notification system.** No trigger events or templates exist for: Contract Sent, Contract Viewed (Salesperson Alert), Contract Signed / Order Confirmation, Sale Alert (Internal), Order Updated, Order Cancelled, Refund Issued, Payment Applied, New Addon Order Confirmation, Booth Confirmation / Booth Upgrade, Bid Won, Sales Re-Assignment, Cart Voided by Salesperson, Workshop/Keynote Confirmation, and the Generic Product Alerts. These templates cannot be seeded meaningfully until the contract, order, payment, booth, bidding, cart-void, workshop, and product-purchase flows are wired to dispatch notifications; QA will not find them in the listing or the trigger dropdown.

8. **A further block of client templates is gated on the deferred capabilities in items 1–3, often in combination with item 7.** Specifically: *Scheduling-blocked* — Contract Reminder, Cart Expiration Reminder, Payment Due, Employee Birthday, Employee Work Anniversary, Event Alert (Day Before / Day Of), Event Photos, Unanswered Product Questions Reminder (including the Booth Builds variant), Workshop/Keynote Reminder; *Scheduling + dynamic-recipient-blocked* — the D&L, ELITeXPO, Venue Manager, GSC, and Electric Orders vendor/logistics reminders; *SMS-blocked* — Unanswered Product Questions (SMS), Workshop Confirmation (SMS), both Generic Product Alert SMS variants, and New Order SMS Confirmation. None of these will appear until the corresponding capability ships.

9. **The client's "PPL Emails" row remains an open scope item.** It is a placeholder describing deadline-driven booth-build-contract emails that the client expects to be "approached separately", and it also depends on the deferred scheduling capability. The seeded PPL templates are the pre-existing pay-per-lead/subscription flows — different emails that share only the PPL tag — so nothing in this release covers that row.

10. **The system ships templates that are not on the client's list, because the catalog covers the existing live production flows.** These are: Admin Welcome and Admin Password Reset (admin-panel counterparts of the client-facing flows), Contact Us Acknowledgment and Contact Us Admin Notification, PPL Subscription Canceled and PPL Subscription Renewal, Invitation Accepted, Lead Assigned (Preview), Daily Lead Summary, Lead Credits Renewed, Low Lead Balance Warning, Order Payment Receipt, and Cart Proposal Updated; three further trigger events (Lead Claimed — Full Details, Lead Claimed by Another Provider, Lead Expired Unclaimed) are seeded for the lead-distribution flows with no template row, and are available in the trigger dropdown for custom templates. QA should expect these rows in the predefined listing even though no client request maps to them.

11. **"Order Payment Receipt" is not the client's "Payment Applied" template.** Despite the similar name, the seeded receipt belongs to the pay-per-lead product-order flow, while the client's Payment Applied is a store-order payment notification that belongs to the separately developed payment flow, which does not send through this module's templates. Treat them as distinct; Payment Applied remains undelivered (see item 7).

12. **The "Cart Proposal Updated" template and its trigger event are included in this release as a late catalog addition.** It supports the cart-proposal flow and was added after the module's original catalog was drawn up, and it behaves like any other predefined template, including the predefined edit restrictions. QA will see 21 trigger events (not 20) and one extra Store-tagged email row in the predefined listing.

### Seeded-data defects

13. **The "Lead Assigned (Preview)" template's seeded copy conflicts with its own trigger.** Its seeded subject/body use placeholders that are not in the trigger's allowed list, so any edit touching the subject or body is rejected with a 400 (edits to other fields, like name or status, still work). Fixing the copy or the trigger's placeholder list requires a data correction.

14. **The "Order Payment Receipt" template body uses conditional blocks the mail engine cannot process.** Its body contains `{{#if}}`-style conditional markup, but the senders only substitute simple tokens, so the conditional markers would render literally in a sent email. Placeholder validation deliberately ignores these control tokens, so editing will not surface the problem either.

15. **"Lead Credits Renewed" is seeded but nothing ever sends it.** The template exists for catalog completeness, but no code path in any of the services dispatches it. QA can view and edit it, but should expect no email from it.

16. **No SMS template rows are seeded, so SMS-specific display states cannot be tested with real data.** All seeded templates are email; combined with custom creation being email-only per the split epics, there is no API path to produce an SMS row. The Channel = SMS visual treatment and the SMS edit restrictions (no subject, locked sending config) are implemented but untestable end-to-end this release.

### Behavioral gaps & limitations

17. **The Active/Inactive status field is stored and editable but is largely unenforced.** The flag can be toggled, filtered on, and is shown in listings, and the live email flows that load templates (the dispatched triggers enumerated in item 21) do skip inactive rows — but seeded templates outside that list have no send path at all, so deactivating them changes nothing observable, and no other behavior (visibility, editability, dropdowns) is driven by the flag. QA should treat it as stored data for now; end-to-end activation semantics arrive with the broader dispatch integration.

18. **Template names are not unique.** Neither create nor edit performs any case-insensitive uniqueness check (the earlier duplicate-rejection on create was removed), so submitting the same payload twice creates two identical rows. QA will see duplicate names save successfully in both flows; whether uniqueness should be enforced is pending a product decision.

19. **Edited sending configuration is never used when emails actually go out.** The live senders take their FROM address from environment configuration and send to the address supplied by the calling code; the stored from-name, reply-to, CC, BCC, and recipient lists are persisted and returned by the API but have zero runtime effect. QA should verify the values round-trip through the API, not that they alter delivered mail.

20. **Sending-configuration edits are additive by design — clearing an override back to the system default is planned but not yet built.** Editing a predefined template's sending configuration creates per-key overrides, and the planned send wiring will read each key individually, falling back to the system default for any key that is not set, so this causes no issue at send time. CC/BCC overrides can already be emptied by saving an empty list; a from-name or reply-to value, however, can currently only be replaced, not cleared — a "clear to default" input (an agreed empty/sentinel value) is planned for a later update, and until it ships a true clear requires a data fix.

21. **A custom template created on a live trigger can be picked up by the real send path — nondeterministically.** The template lookup used by the live email flows does not distinguish predefined from custom and applies no ordering, so an active custom template sharing a dispatched trigger may shadow the predefined template for that trigger unpredictably; conversely, custom templates on any non-live trigger are never sent at all (no general dispatcher exists). This is a defect, not a design limitation — it is tracked in the project's known-issues register with a proposed fix (restrict the live-flow template lookup to predefined templates only), which is planned to be delivered together with the upcoming scheduling logic, and until that fix ships QA should avoid creating active custom templates in shared environments on the dispatched (live) triggers: `welcome_email`, `forgot_password`, `exhibitor_welcome_admin_created`, `cart_updated_notification` (admin backend); `welcome_email_exhibitor`, `exhibitor_forgot_password`, `company_user_invitation`, `invitation_accepted_to_exhibitor`, `contact_us_acknowledgment`, `contact_us_admin_notification`, `ppl_subscription_canceled` (exhibitor backend); `ppl_order_confirmation`, `ppl_subscription_renewal`, `ppl_product_order_payment` (external API); `lead_assigned_preview`, `lead_daily_summary`, `low_balance_warning` (background worker).

22. **The template record carries no "last modified by".** The model stores only the last-updated timestamp; the modifier's identity must be derived from the central admin audit-log endpoint using the entity-type and entity-id filters, and there is no per-template audit endpoint. The listing story that expects a modified-by column is pending a wording update.

23. **Audit snapshots are summaries, and audit permanence is conditional.** The create/delete audit records capture a seven-field summary (name, trigger, channel, tag, language, status, FROM address) — not the subject, body, or recipient lists — so a deleted template's full content cannot be reconstructed from the audit trail. The audit store is append-only with no delete API today, but a three-month retention setting and a database cleanup routine already exist (currently un-wired), so "permanent" holds only until retention is switched on.

24. **The Type vocabulary is Store / Internal / Vendor / Product / PPL / System — there is no "Event" type.** Some stories list "Event" as a type, but the implementation (and other stories) use "System" instead, so filtering or creating with type Event fails validation. This discrepancy stands until the story wording is reconciled.

25. **The Channel column never shows "Both".** Every template is strictly single-channel (Email or SMS); the listing story's "or both" has no counterpart in the data model and the business analysts have been asked to drop it. QA will never encounter a Both value in the API or UI.

26. **The delete endpoint has no backing user story.** Delete exists for custom templates per the agreed design document (full CRUD on custom), but neither set of stories contains a delete row — the predefined side is edit-only by design and the custom side simply omits it. QA testing deletion has nothing to trace it to until the business analysts add a story or formally confirm the design-doc justification.

27. **The "event selector" wording in the custom listing story will be updated by the business analysts.** It references the dropdown the frontend needs to show for placeholders — selecting a trigger event surfaces the placeholders available to it — not a listing-page filter or create shortcut. The supporting API already exists (the trigger-event list returns each event with its allowed placeholders), so QA should expect the placeholder dropdown on the create/edit flows and no event control on the listing page.

28. **The API enforces validation rules no story specifies.** QA will hit length caps on names/subjects/search, recipient array limits (1–50 TO, up to 50 CC/BCC), email-format checks, rejection of unknown placeholders and unknown config keys with field-specific 400s, and the email-only channel message; search also matches the subject field beyond the stories' wording, and TO recipients are intentionally not de-duplicated (only CC/BCC are). These rules are tracked for business-analyst confirmation so stories and implementation converge.

29. **Concurrent edits are last-write-wins.** There is no version or precondition mechanism on the edit endpoint, so two admins editing the same template will silently overwrite each other field-by-field. The audit trail records both edits, but the API gives no conflict warning.

30. **Behavior for missing placeholder values at send time is undecided.** When a live flow sends a template and a token has no value, the token is left literally in the delivered email — there is no fallback rule (blank, default text, or block-send), which remains subject to R&D. QA testing live flows may see raw `{{token}}` text in mail when data is absent.

31. **The hardened id validation covers this module's template routes plus the booth-agreement `:id` routes, not the whole admin API.** Hex, exponent, and padded id strings now return 400 on the template GET/PUT/DELETE routes and on the three booth-agreement `:id` routes (which share the same guarded raw-param id pipe), but other admin routes still silently coerce such values to the wrong row. That wider gap — roughly 60 admin routes still on the plain `ParseIntPipe` — is recorded as a separate, undecided repo-wide item in the 2026-06-12 scope note in `EMAIL_SMS_API_CHANGELOG.md`; QA may still observe the coercion elsewhere in the admin panel.

### Data & environment notes

32. **Seeders must be re-run in every environment before the module is usable.** The four new template permission groups (View/Create/Update/Delete) and the permissions behind the trigger-events and allowed-FROM-domains endpoints only exist after a seed run — until then even Admin users get 403 on the new endpoints and the permissions are missing from the role screen. Non-admin roles additionally need the new groups enabled per role.

33. **The template seeder is create-only, which changes how copy fixes ship.** Re-running the seeder skips existing rows so admin edits survive, but the trade-off is that copy changes to already-seeded environments can no longer be delivered via the seeder — they require a one-off migration or a manual admin edit. Plan content corrections accordingly.

34. **The API response shapes changed in breaking ways for any existing consumer.** List rows are now lean (no subject/body/language/created-at — use the detail endpoint), ids are numbers instead of strings, omitted optional fields come back as explicit nulls, sorting changed to most-recently-updated first, and create/edit now return the full detail shape. Frontend and any integration consuming these endpoints must be updated in step with this release.

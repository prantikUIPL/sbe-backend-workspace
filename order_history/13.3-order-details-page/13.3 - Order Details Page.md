## Summary

Read-only exhibitor Order Details page. The core data is well-backed: the exhibitor cart checkout write-path (exhibitor-backend-api cart.service.ts:1229-1305 + cart.helpers.ts materializeCartToOrder) writes Order (subtotal, coupon_amount, setup_fees, cleaning_fees, total_savings, total, paid_amount=0, billing), OrderItems carrying cart_item_type (booth/sponsorship/addon/fee), description, quantity, unit_price, amount and parent_order_item_id, plus OrderAgreement (signer, signature URL, terms_accepted, signed_at) and a processing PaymentTransaction; the external-api webhook (webhook.service.ts:2042-2098 + buildPtSuccessUpdate:2187-2201) flips the order to paid, sets paid_amount/charge id/paid_at, and creates Invoice + InvoiceLineItem tree for cart-flow orders. So a net-new GET order-details endpoint, an invoice-PDF endpoint (a pdfkit renderer already exists in ppl.service.ts) and an agreement download are all Deliverable on built primitives, with ownership enforced via existing auth guards + company_id scoping. Booth Size is now Deliverable: it is joinable structured data (booth OrderItem.product_id -> Product.length/width, the canonical 10x10), not just a name snapshot. Gaps: Gift Certificate redemption has a model (GiftCertificateRedeem) but NO code writes it - admin only reads it (Blocked); booth Number has no column/write-path anywhere (Not Deliverable); and the 'download signed agreement' option only has the captured signature image + acceptance metadata, no composed agreement document is generated (Partial). Per the story note, Booth Type, Booth Upgrade, salesperson Payment Schedule visibility, and Onsite Contact belong to other stories (Out of Scope).

## Open Questions

- **[OPEN — for team discussion]** Booth Number: does the business assign a physical stall/booth number that must be displayed, or is 'Booth Number' intended to mean the booth product/SKU name? No model or write-path currently captures a stall number anywhere.
  - **Current direction (2026-07-01):** — (pending). No stall-number data exists in any repo/schema, so 13.3-c stays **Not Deliverable** until either the business confirms it means the booth product/SKU name (then it becomes a passthrough of the existing OrderItem name) or a booth-number assignment write-path is scoped in another epic.
- **[OPEN — for team discussion]** Download signed agreement: is a full composed agreement document (terms text + signature) expected, or is the captured signature image plus acceptance metadata (terms_version, signed_at, IP) sufficient? Only the latter is persisted today.
  - **Current direction (2026-07-01):** — (pending). Only the signature image + acceptance metadata are persisted; a composed agreement PDF (terms merged with signature) is net-new and depends on the agreement-terms content source. Leaning toward shipping the metadata/signature download first (13.3-v Partial) and deferring the composed document pending confirmation.
- **[OPEN — for team discussion]** Multi-show orders: a single cart/order can span multiple shows (cart response groups items by show), so should the header show one event or list per-show event/city/dates?
  - **Current direction (2026-07-01):** — (pending). Since an order can span multiple shows, the leaning is to return a per-show list of event/city/dates in the header rather than a single event; awaiting team confirmation before fixing the DTO shape.

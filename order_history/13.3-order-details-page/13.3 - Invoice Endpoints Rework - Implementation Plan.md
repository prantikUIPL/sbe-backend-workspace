# SBE-1146 Implementation Plan ‚Äî Exhibitor Invoice Endpoints Rework (v2 ‚Äî Approved)

**Scope:** 1 session. Contract revision of shipped requirement **13.3-s** (Download Invoice PDF): the
download endpoint stops resolving "the order's latest invoice" and instead serves **the invoice the
caller names**, and a **new invoice-list endpoint** supplies the FE's invoice ids. The old order-id
download route is **removed outright** (user decision 2026-07-16 ‚Äî "the existing endpoint will not be
required"). v1 covered the download only; v2 folds the list endpoint into the same session (user
decision at plan review).

**Source:** `order_history/13.3-order-details-page/13.3 - Order Details Page.md` ‚Üí 13.3-s (shipped
2026-07-05, PR #300, Option B) ‚Üí this revision. Ticket: SBE-1146 (13.3 has no own Jira key; it rides
the Order History epic ticket).

## Context

**Why:** 13.3-s shipped as a single-button MVP ‚Äî `GET /orders/:orderId/invoice` returns only the
**latest** persisted invoice (`paid_at DESC, id DESC`). Split orders persist N invoices, so the
exhibitor can never retrieve the older ones. Latest-only is also the outlier contract: PPL exhibitor
(`GET /ppl/orders/:id/invoices` + `GET /ppl/invoices/:id/download`) and admin 24.5-g
(`GET /orders/:id/invoices/:invoiceId/invoice`) are both per-invoice. This rework aligns the
exhibitor surface with both precedents: a list to discover invoices, a by-id download to fetch each.

**What:** (1) Replace the route `GET /orders/:orderId/invoice` with `GET /orders/invoices/:id` on the
same `OrdersController`; change the service lookup from latest-per-order to by-id (ownership-404
discipline unchanged); delete the latest-selection ordering; rewrite Swagger. (2) Add
`GET /orders/:orderId/invoices` returning the order's invoice headers (`created_at DESC`, encrypted
ids, empty array when none). No renderer, cache, upload, schema, or admin-repo changes.

## Design Decisions

### D1: Flat download route `GET /orders/invoices/:id`, wire param `id`

Locked with the human 2026-07-16 (flat over admin-mirror nested; over `GET /orders/:invoiceId/invoice`).
Rationale: the path segment names the entity, so the param's meaning is self-evident; avoids the
"invoice vs invoices" same-position/different-entity trap that produced SBE-1590; matches PPL's
`/ppl/invoices/:id/download` and admin's per-invoice semantics. Wire param is **`id`** (not
`invoiceId`): the shared id-crypto decrypts `id`/`*_id` by base rule, and `invoiceId` is deliberately
absent from `ENCRYPTABLE_ID_SCALAR_ALIASES` ‚Äî using `:id` means **zero shared-crypto change** and
exactly mirrors the PPL controller (`@Param('id', ParseIntPipe) invoiceId: number`). No route
conflict: `invoices/:id` is two segments; the `:orderId` details route is one. Declared above the
param routes as route-hygiene.

### D2: Download lookup by id; ownership-404 discipline preserved verbatim

`loadLatestInvoice(orderId, companyId)` ‚Üí `loadInvoice(invoiceId, companyId)`:
`where { id: invoiceId, deleted_at: null, order: { is: { company_id: companyId, order_type: product,
deleted_at: null } } }` ‚Äî the `orderBy` line (the entire "latest" behavior) is **deleted**. Foreign,
unknown, soft-deleted, or non-product-order invoice ‚Üí 404 `'Invoice not found'` (drop "for this
order"), never leaking another company's data. `findFirst` retained (compound scoping on a unique id).

### D3: Old route removed outright ‚Äî hard breaking change, no shim

User decision: the order-id form has no future consumer. No deprecation period, no 410 handler ‚Äî
after this ships, `GET /orders/:orderId/invoice` is an unmatched route (generic 404). FE switches in
the same deploy. Shipped staging-first (branch cut from `origin/staging`; `src/orders/` verified
byte-identical staging‚Üîdev), dev twin cherry-picked after the staging PR merges ‚Äî same dual-track
flow as SBE-1592.

### D4: List endpoint mirrors the PPL shape; product-gated; empty array over 404-when-none

`GET /orders/:orderId/invoices`: ownership gate first (`order.findFirst { id, company_id,
order_type: product, deleted_at: null }` ‚Üí 404 `'Order not found'`; product-gated to match the
download ‚Äî subscription/PPL invoices are the PPL module's surface), then
`invoice.findMany { order_id, deleted_at: null }`, `orderBy { created_at: 'desc' }`, header select
`id, invoice_number, status, total, paid_at, created_at`. An owned order with 0 invoices returns
**200 `{invoices: []}`** (matches PPL; unpaid orders simply have no invoices yet ‚Äî
`can_download_invoice` already signals this to the FE). Ids are encrypted on the wire by the global
interceptor and drop straight into D1's `:id`.

### D5: Both methods live on `OrderInvoiceDocumentService`; everything downstream untouched

The service already has prisma + `resolveExhibitorCompanyId`; the list is a sibling read
(`listForOrder(exhibitorId, orderId)`), no new file/provider. `INVOICE_INCLUDE`, `renderPdf`,
`uploadAndCache`, the cache-first `invoice_pdf_url` check, and the S3 key are already keyed to the
invoice row ‚Äî zero changes. Download response stays `{ url }` (`OrderInvoiceUrlResponseDto` shape
unchanged; only doc text updates).

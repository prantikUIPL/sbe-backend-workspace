## Gate Contract

- `GET /orders/:orderId` returns HTTP 200 with `{ success, message, data: {…OrderDetailsResponseDto…}, timestamp }` for an authenticated exhibitor who owns the order.
- **Ownership (13.3-x):** the order is looked up with `where: { id, company_id, deleted_at: null }`, `company_id` resolved server-side; a foreign or missing order → **404** `NotFoundException` (never leaks another company's data).
- **Header (13.3-a, -b):** `order_number`, `order_date`, derived `payment_status`, and a **per-show list** `shows[]` (`{ id, title, city, date }`), reusing 13.2's `buildOrderShows`.
- **Booth section (13.3-d, -e):** `booths[]`, one per booth/workshop-pavilion line, each with `size` (e.g. `"10x10"`), `unit_price`, `quantity`, and `subtotal` (line + bundled children).
- **Add-Ons (13.3-h, -aa)** and **Sponsorships (13.3-i, -aa):** arrays of `{ description, quantity, unit_price, amount }`.
- **Financial Summary (13.3-j, -k, -l, -m, -n, -o, -y, -z):** `subtotal`, conditional `coupon_code`/`coupon_amount`, **`gift_certificate_code`/`gift_certificate_amount`** (13.3-l, added 2026-07-05 — mutually exclusive with the coupon line), `cleaning_fee`/`setup_fee`/`total_savings`, `total`, `total_paid`, `currency` — nullable fields present **only when applicable**; values are stored snapshots that reconcile.
- **Payment Details (13.3-p, -q, -r):** `payments[]` of **settled** transactions only, each `{ method: {brand,last4,type}|null, paid_at, reference, amount }`.
- **Agreement (13.3-u):** `agreement` = `{ accepted, accepted_at, signer_name, terms_version }` or `null`.
- Detail-only pure helpers unit-tested in isolation; all quality gates green (lint, typecheck, jest).
- **Not shipped** in the original cut (Blocked/Deferred — see Deferred): booth number/type/upgrade, gift-certificate line, download-invoice, payment schedule, download-signed-agreement, onsite contact. _Update 2026-07-05 (see [addendum](#post-approval-delivery-update-2026-07-05)): **download-invoice (13.3-s) and gift-certificate line (13.3-l) have since shipped**; **download-signed-agreement (13.3-v) is Deliverable via the existing endpoint**. ## Context

**Why:** Story 13.2 shipped the order *listing* (`GET /orders`) whose View action navigates to a details screen that does not yet exist. There is no endpoint that returns a single order's complete, read-only breakdown (header, booth, add-ons, sponsorships, financial summary, payment details, agreement acceptance). Without 13.3 the listing's primary action dead-ends.

**What:** Extend the existing `OrdersModule` in `exhibitor-backend-api` with one endpoint, `GET /orders/:orderId`, backed by a new `OrderDetailsService.getOrderDetails(exhibitorId, orderId)`. It reads a single company-scoped `Order` plus its `orderItems` (classified by `cart_item_type` into booth / add-on / sponsorship), `agreement`, and `paymentTransactions`, joins the coupon code through the cart and payment-method brand/last4 through a keyed lookup, and assembles a sectioned response. Read-time derivations reuse 13.2's pure helpers (`deriveOrderPaymentStatus`, `buildOrderShows`); detail-specific assembly lives in new pure helpers.

## Design Decisions

### D1: Extend the existing `orders` module — new route on `OrdersController`, dedicated `OrderDetailsService`

Add `@Get(':orderId')` to the existing `OrdersController` (keeps `@Controller('orders')` cohesive) delegating to a **new `OrderDetailsService`** provided by `OrdersModule`. Rationale: the listing service (`OrdersService`) stays stable/untouched; the details assembly (~150 lines: classification, joins, section building) gets its own focused home rather than bloating the listing service. Both services inject only `PrismaService`.

### D2: Exhibitor host conventions (inherited from 13.2)

Identical to the 13.2 endpoint: class-level `@UseGuards(JwtAuthGuard)` + `@ApiBearerAuth('JWT-auth')`; `@Req() req: AuthedRequest` → `req.user.id`; numeric path param via `@Param('orderId', ParseIntPipe)` (repo convention — see `agreement.controller.ts:182`); wrap via `successApiResponse(msg, data)`; Swagger via `@ApiOkWrappedResponse` + `@ApiAuthResponses` + `@ApiNotFoundResponse`.

### D3: `select`-constant + `GetPayload` (13.2 style), security-minimal

Define `ORDER_DETAILS_SELECT satisfies Prisma.OrderSelect` and `type OrderDetailsRow = Prisma.OrderGetPayload<{ select: typeof ORDER_DETAILS_SELECT }>`. Select **only** presentation fields — **no** billing address, no `stripe_customer_id`, no `signature_data`/`ip_address`/`user_agent`. Nested selects: `cart.coupon.code`; `orderItems` (`id, parent_order_item_id, cart_item_type, description, quantity, unit_price, amount, product{length,width}, showProduct.show{…city}`); `agreement` (`terms_accepted, signer_first_name, signer_last_name, signed_at, terms_version`); `paymentTransactions` (`status, amount, paid_at, stripe_charge_id, stripe_payment_intent_id, stripe_payment_method_id`).

### D4: Ownership via a single scoped `findFirst` (13.3-x)

`companyId = resolveExhibitorCompanyId(prisma, exhibitorId)`, then `order.findFirst({ where: { id: orderId, company_id: companyId, deleted_at: null }, select: ORDER_DETAILS_SELECT })`. `null` → `throw new NotFoundException('Order not found')`. A foreign order and a non-existent order are indistinguishable to the caller (no enumeration leak).

### D5: Line classification by `cart_item_type` → `booths[]` / `add_ons[]` / `sponsorships[]` (13.3-d,-e,-h,-i,-aa)

Pure `classifyOrderItems(items)`:
- **booth** or **workshop_pavilion** → `booths[]`. Each booth: `size` from its `product.length`×`product.width` via `formatBoothSize` (`10.00,10.00 → "10x10"`, trailing-zero trimmed; `null` if either missing), `unit_price`, `quantity`, and `subtotal = booth.amount + Σ children.amount` (children = items whose `parent_order_item_id === booth.id`, i.e. bundled/default-included lines & fees). Multi-show orders yield multiple booth lines → an **array** (consistent with the per-show header, 13.3-b).
- **addon** → `add_ons[]`; **sponsorship** → `sponsorships[]` — each `{ description, quantity, unit_price, amount }`.
- **booth_setup_fee / booth_cleaning_fee** and null `cart_item_type` (subscription/ppl_addon lines) are **not** emitted as sections (fees surface in Financial Summary from the `Order` snapshots). Non-product orders simply yield empty section arrays.

### D6: Financial Summary from `Order` snapshots, conditional display (13.3-j,-k,-m,-n,-o,-y,-z)

`buildFinancialSummary(order, couponCode)` reads stored snapshots (no recompute — 13.3-z): `subtotal`, `total`, `total_paid = paid_amount`, `currency` always; `coupon_amount`+`coupon_code`, `cleaning_fee`, `setup_fee`, `total_savings` emitted as **`null` when zero/absent** (13.3-y conditional display). **Coupon code** comes from `cart.coupon.code` (the `Order.coupon_code` column is not populated at checkout); `null` when no cart/coupon. Gift-certificate line is **excluded** (13.3-l Blocked).

### D7: Payment method via a second keyed lookup; `payments[]` = settled transactions only (13.3-p,-q,-r)

`PaymentTransaction` has **no** Prisma relation to `PaymentMethod` — they share only `stripe_payment_method_id` + `company_id`. So after the order load, collect the distinct `stripe_payment_method_id` of **settled** transactions and run one `paymentMethod.findMany({ where: { company_id, stripe_payment_method_id: { in } }, select: { stripe_payment_method_id, brand, last4, type } })` (**no `deleted_at` filter** — a removed card must still render brand/last4, per the 13.3-p edge note); build a `Map`. `buildPaymentRecords(transactions, methodMap)` emits **only settled** transactions (`status = succeeded`, i.e. an actual payment with `paid_at`/`stripe_charge_id`), each `{ method: {brand,last4,type}|null, paid_at, reference: stripe_charge_id ?? stripe_payment_intent_id, amount }`. **Scheduled/future installments are excluded** — that is the Payment *Schedule* (13.3-t, Blocked), not payment *history*.

### D8: Reuse 13.2 helpers for the header; detail helpers are pure and live in `order-details.helpers.ts`

`payment_status` (13.3-a) via `deriveOrderPaymentStatus(paid_amount, total)` and `shows[]` (13.3-b) via `buildOrderShows(orderItems)` — imported from `orders.helpers.ts`, no duplication. New detail derivations (`formatBoothSize`, `classifyOrderItems`, `buildFinancialSummary`, `buildPaymentRecords`, `buildAgreement`) are pure (Decimal→number only) and unit-tested in isolation.

## Public API Contract

### Caller Contract

`GET /orders/:orderId` — authenticated exhibitor (JWT cookie). Returns the order **only if** it belongs to the caller's company (derived server-side); otherwise 404. `:orderId` is a positive integer (`ParseIntPipe`).

### Consumer Responsibilities

- Render `payment_status` as the header badge (e.g. `paid_in_full` → "Paid in Full"); use raw `status` for canceled/failed/refunded states.
- Treat any `null` financial field and any empty section array as "not applicable" (hide the row) — 13.3-y.
- `payments[]` is **paid history**, not a schedule; the payment schedule and download buttons are separate (blocked) features and are absent here.

## Security Considerations

- **Ownership (13.3-x):** every read is filtered by the server-derived `company_id`; `:orderId` alone is never trusted. Cross-tenant test asserts exhibitor B gets 404 for exhibitor A's order. The payment-method lookup is **also** scoped by `company_id` (a `pm_id` from the order can only match that company's cards).
- **Minimal projection:** `select` omits billing address, `stripe_customer_id`, and the agreement's `signature_data`/`ip_address`/`user_agent` — only presentation fields leave the service.
- **No recompute:** monetary values are read straight from `Order`/`OrderItem` snapshots (13.3-z), avoiding drift.
- **Auth:** class-level `JwtAuthGuard`; unauthenticated → 401 (`@ApiAuthResponses`). Input `:orderId` hardened by `ParseIntPipe`.

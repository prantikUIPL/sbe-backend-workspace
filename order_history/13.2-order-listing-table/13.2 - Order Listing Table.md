# Story 13.2 — Order Listing Table

**Epic:** Order History

**Confluence:** [Order History → Story 13.2](https://unifiedinfotech.atlassian.net/wiki/spaces/SBE/pages/3858137106/SBE+-+Exhibitor+Store+Application#Order-History) _(page section; Confluence has no per-requirement anchor)_

**Build Status:** 13 requirement(s) to implement here (filter Build Focus = 'Implement here'). 0 covered elsewhere, 1 out of scope.

## Summary

Story 13.2 asks for a paginated, ownership-scoped table of an exhibitor's orders showing Order ID, Order Date, derived Payment Status (Paid in Full / Partially Paid / Unpaid), Associated Shows (multi-show grouped), and Total Amount, plus per-row View / Invoice / Pay actions. Every primitive this needs is already BUILT: the Order model with paid_amount/paid_in_full_at/status; a real Stripe webhook write-path (external-api-service webhook.service.ts) that bumps paid_amount and flips status to partially_paid/completed via computeNewOrderStatus; order->orderItems->showProduct->show joins plus buildOrderGroup grouping (purchased-booths.service); a paginated company-scoped order listing pattern (ppl.service listOrderHistory); and a pdfkit render + S3 signed-URL invoice download path. Per-installment Paid/Unpaid/Overdue are also fully derivable: PaymentTransaction.due_date is written by a real path (payments.service split-schedule) and status by checkout/cron/webhook, so Overdue = due_date<now AND status!=succeeded is a pure derivation just like the order-level status. The net-new listing + invoice endpoints are therefore Deliverable. The only out-of-scope piece is the Pay-flow EXECUTION (payment-summary screen, salesperson-set method display, card capture), which the spec explicitly delegates to the Credit Card Payment Workflow / Payment Schedule Visibility stories; only the Pay-action availability flag is in scope here.

## Open Questions

- **[RESOLVED 2026-06-30]** Does 'all the Exhibitor's orders' include PPL subscription/ppl_addon orders (which have no showProduct and would render an empty 'Associated Shows' column), or is the listing limited to product/booth (show) orders? Today these are listed by two different, non-overlapping paths — ppl.service.listOrderHistory filters to subscription/ppl_addon only and explicitly EXCLUDES product orders, while purchased-booths filters to product/show orders only — so the unified scope of this new listing is a genuine product decision.
  - **Decision:** `GET /orders` spans **all three order types** (`product`, `subscription`, `ppl_addon`) from the single `orders` table — it does NOT narrow by `order_type`. All orders live in one `Order` model (`admin-backend-api/prisma/schema.prisma:1540`, `order_type` enum at :1545); the existing PPL and purchased-booths pages only differ by their `order_type` filter, so the unified listing simply relaxes that predicate. Consequence: 'Associated Shows' renders only for `product` orders (which carry `orderItems → showProduct → show`) and is an **empty array `shows: []`** (never null) for `subscription`/`ppl_addon` orders, which the row DTO returns gracefully.

- **[OPEN — for team discussion]** Should abandoned `pending` orders appear in the listing? A `pending` Order row is written at place-order time (before the Stripe redirect); if the exhibitor backs out it stays `pending` forever. The existing PPL history deliberately excludes `pending` and shows only terminal/active states.
  - **Current direction (2026-06-30):** **Include `pending` rows** in `GET /orders` so in-flight/current orders surface (consistent with 13.1's "past AND current" mandate and the Pay action on payable orders). Flagged for team confirmation because this also surfaces never-paid abandoned-checkout rows, which may add noise; team to decide whether to keep, filter by an age cutoff, or exclude pure abandoned-checkout pending orders.

- **[OPEN — for team confirmation · WORK DEFERRED]** What does the "Invoice" action mean for the listing — a **formal tax invoice/receipt for money actually paid**, or an **order document/summary** the exhibitor can download regardless of payment state? This determines whether 13.2-g serves the persisted `Invoice` rows or generates an order-level document on the fly. Current invoice generation is **PPL/subscription-only**: the Stripe webhook (`external-api-service/webhook.service.ts:679,762,2048,2225`) persists `Invoice`+`InvoiceLineItem` rows **only on successful payment** (unpaid order → 0 invoices; split order → N invoices), and the **single** PDF renderer (`exhibitor-backend-api/ppl.service.ts:985 renderInvoicePdf`, pdfkit, exposed at `GET /ppl/invoices/:id/download`) is hardcoded subscription-shaped (`package_name`/`subscription_status`/`billing_frequency`/recurring-charge message) and cannot render product/booth orders as-is. There is no product-order invoice renderer anywhere.
  - **Options:** **(A)** generate an order-level invoice/summary PDF from `Order`+`OrderItems` (works for every order & payment state; satisfies "Invoice available for every order"; generalizes the pdfkit renderer) vs **(B)** serve only persisted `Invoice` rows (faithful to accounting/QuickBooks but unpaid → 404, split → latest-vs-all ambiguity, still needs a product-shaped renderer).
  - **Decision (2026-06-30):** **DEFER the Invoice endpoint work (13.2-g, `GET /orders/:orderId/invoice`)** until the team confirms invoice semantics (A vs B). The listing endpoint (`GET /orders`) and all other rows/actions proceed now; the Invoice **action-availability flag** and the invoice PDF endpoint are held pending this answer. See **Deferred Work** below.

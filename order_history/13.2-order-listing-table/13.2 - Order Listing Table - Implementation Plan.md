# SBE-13.2 Implementation Plan — Exhibitor Order Listing Table (v1 — Draft)

**Scope:** 1 session. A net-new read-only `GET /orders` listing endpoint in `exhibitor-backend-api`, serving the Order History epic's listing story (13.2) and, by sharing the same endpoint, the container story (13.1).

**Source:** Story 13.2 feasibility — `order_history/13.2-order-listing-table/13.2 - Order Listing Table.md` (+ `.xlsx`).

---

## Gate Contract

- `GET /orders` returns HTTP 200 with `{ success, message, data: { data: [...], meta, isEmpty }, timestamp }` for an authenticated exhibitor.
- Rows are scoped to the caller's `company_id` only — a second exhibitor's orders never appear (ownership test green).
- Listing is scoped to **`product`** (booth) orders only (`order_type = product`, SBE-1621) and **includes `pending`** orders (no status narrowing); `subscription`/`ppl_addon` orders are served by `/ppl/orders`, not here.
- Each row carries: `id`, `order_number`, `order_date`, `order_type`, `status`, derived `payment_status` (`paid_in_full` | `partially_paid` | `unpaid`), `is_overdue`, `total`, `currency`, `shows[]` (empty array for non-`product`), and `can_pay`.
- `meta` = `{ page, limit, total, totalPages, hasNext, hasPrev }`; `isEmpty === true` when the exhibitor has no orders.
- Pure derivation helpers (`deriveOrderPaymentStatus`, `deriveIsOverdue`, `deriveCanPay`, `buildOrderShows`) unit-tested in isolation.
- All quality gates green (lint, typecheck, jest).
- **No** invoice endpoint/flag is shipped (deferred — see Deferred).

## Context

**Why:** The exhibitor app has no unified order history. Today two non-overlapping read paths exist — `ppl.service.listOrderHistory` (subscription/ppl_addon only) and `purchased-booths.service` (product/show only) — and neither presents a single "all my orders" table with derived payment status and per-row actions. Without 13.2 the Order History page (13.1) and the Order Details navigation (13.3) have no listing to hang off; both are marked blocked-on-13.2.

**What:** A new `OrdersModule` in `exhibitor-backend-api` exposing one endpoint, `GET /orders`, backed by `OrdersService.listOrders(exhibitorId, query)`. It returns a paginated, company-scoped list over the single `Order` table, scoped to `product` (booth) orders (`order_type = product`, SBE-1621), with read-time derivations: order-level payment status (from `paid_amount` vs `total`), an `is_overdue` rollup (from `PaymentTransaction.due_date`/`status`), grouped associated `shows[]` (from `orderItems → showProduct → show`), and a `can_pay` action flag. View is satisfied by returning each order's `id`; the details screen and invoice download are other deliverables.

## Design Decisions

### D1: New `orders` module, read-only, Prisma-only deps

A dedicated `exhibitor-backend-api/src/orders/` module (`OrdersModule` → `OrdersController` + `OrdersService`), registered in `app.module.ts`. The endpoint is read-only, so the service injects **only `PrismaService`** — no audit service, no upload service (those belong to write/invoice paths). Mirrors booth-agreements file layout (kebab-case, co-located specs, single `dto/index.ts` barrel) but uses **exhibitor host conventions**, not admin ones (see D2).

### D2: Exhibitor host conventions (not admin's)

- Auth: class-level `@UseGuards(JwtAuthGuard)` + `@ApiBearerAuth('JWT-auth')` (NOT admin `@Permissions`).
- Current user: `@Req() req: AuthedRequest` where `type AuthedRequest = Request & { user: { id: number } }`; pass `req.user.id` as `exhibitorId`.
- Response envelope: wrap via `successApiResponse(message, data)` → `ApiResponseDto<T>` (`exhibitor-backend-api/src/common/utils/api-response.util.ts`, `.../common/dto/api-response.dto.ts`). Swagger via `@ApiOkWrappedResponse(Dto, msg)` + `@ApiAuthResponses()`.
- Query DTO extends the exhibitor `PaginationDto` (`.../common/dto/pagination.dto.ts`, which already carries inline `page=1`/`limit=10` defaults + `PaginationMetaDto`).
- Ownership: reuse the shared `resolveExhibitorCompanyId(prisma, exhibitorId)` helper (`.../common/helpers/resolve-exhibitor-company.helper.ts`) — already enforces `deleted_at: null, status: true` and throws `NotFoundException`.

### D3: `select`-constant + `GetPayload` serialization (booth-agreements style)

Define a frozen `ORDER_LIST_SELECT satisfies Prisma.OrderSelect` and derive the row type `type OrderListRow = Prisma.OrderGetPayload<{ select: typeof ORDER_LIST_SELECT }>`. Select only what the row needs: scalar columns (`id, order_number, order_type, status, total, currency, paid_amount, created_at`) + `orderItems` (nested `showProduct.show_id` + `showProduct.show` with its `city` — `parent_order_item_id`/`show_product_id` are intentionally **not** selected; see D4 for why the parent walk is unnecessary) + `paymentTransactions` (`status, due_date` only). No `include` of unused relations.

### D4: Read-time derivations are pure functions in `orders.helpers.ts`

Three order-level derivations + one grouping, all pure and unit-testable, kept out of the Prisma orchestration so 13.3 can reuse them:

- **`deriveOrderPaymentStatus(paidAmount, total) → 'paid_in_full' | 'partially_paid' | 'unpaid'`**: `paidAmount <= 0` → `unpaid`; else `paidAmount >= total` → `paid_in_full`; else `partially_paid`. (Pure money derivation — does not consult `status`, which is overloaded with `failed`/`refunded`/`canceled`. Confirmed: there is no payment-status enum; `Order.paid_amount`/`total` are the authoritative written fields, bumped by the Stripe webhook.)
- **`deriveIsOverdue(paymentTransactions, now) → boolean`**: `some(pt => pt.due_date < now && pt.status NOT IN ('succeeded','refunded','canceled'))`. There is **no `overdue` enum value** (`PaymentTransactionStatus` = scheduled/processing/succeeded/failed/canceled/refunded) — overdue is a pure derivation. Satisfies 13.2-d at the listing level; the order-level rollup (13.2-e) is already covered by `partially_paid`/`unpaid`.
- **`deriveCanPay(paymentStatus, status) → boolean`**: `paymentStatus !== 'paid_in_full' && status NOT IN ('canceled','refunded')`. Emits 13.2-h's Pay-availability flag; pay-flow destination is out of scope.
- **`buildOrderShows(orderItems) → OrderShow[]`**: collect the **distinct** shows directly from each item's `showProduct.show`, deduped + sorted by show id. **Simplification vs. an explicit `parent_order_item_id` walk:** a default-included add-on may carry a null `show_product_id`, but it always hangs off a parent booth line in the *same order* whose `showProduct.show` already contributes that show — so distinct-collection yields the identical show set without selecting or walking `parent_order_item_id`. (The lone case a parent walk could not recover either is a booth line whose `showProduct` was `SetNull`-deleted, where no `show_id` survives on parent *or* child.) Returns **`[]`** for `subscription`/`ppl_addon` orders (no `showProduct`), never `null`. Produces the same displayed-show set as `purchased-booths.service` `buildOrderGroup`.

### D5: Listing scope filter — minimal `where`

Per resolved decisions (as amended by SBE-1621): `where = { company_id, order_type: product, deleted_at: null }` — **`product`-only** (`order_type = product`) with **no status narrowing** (`pending` still included). Optional `search` query param matches `order_number` (case-insensitive `contains`) for the "many orders" UX (13.2-l); kept minimal. `orderBy: { created_at: 'desc' }`. Pagination via `Promise.all([findMany({ skip, take }), count()])`, then hand-built `meta` + `isEmpty: total === 0`.

### D6: Row DTO carries raw `status` AND derived `payment_status`

The row returns both the raw `status` enum (so the FE can badge `canceled`/`failed`/`refunded`) and the derived 3-bucket `payment_status` (the story's Payment Status column). `shows` is always an array. `can_pay` is the only action flag in this cut (View = the `id`; Invoice flag deferred).

## Public API Contract

### Caller Contract

`GET /orders?page=&limit=&search=` — authenticated exhibitor (JWT cookie). Returns the caller's own orders only; `company_id` is derived server-side from `req.user.id`, never from client input.

### Response shape

```jsonc
// ApiResponseDto<OrderListResponseDto>
{
  "success": true,
  "message": "Orders retrieved successfully",
  "data": {
    "data": [
      {
        "id": 123,
        "order_number": "ORD-2026-00123",
        "order_type": "product",            // product | subscription | ppl_addon
        "order_date": "2026-06-01T10:00:00.000Z",
        "status": "partially_paid",          // raw OrderStatus
        "payment_status": "partially_paid",  // derived: paid_in_full | partially_paid | unpaid
        "is_overdue": false,                 // derived from payment transactions
        "total": 1500.00,
        "currency": "usd",
        "shows": [                            // [] for subscription/ppl_addon
          { "id": 7, "title": "Small Business Expo NYC", "city": "New York", "date": "2026-09-10T00:00:00.000Z" }
        ],
        "can_pay": true                       // derived; View = use `id`; Invoice flag deferred
      }
    ],
    "meta": { "page": 1, "limit": 10, "total": 1, "totalPages": 1, "hasNext": false, "hasPrev": false },
    "isEmpty": false
  },
  "timestamp": "2026-06-30T12:00:00.000Z"
}
```

### Consumer Responsibilities

- Treat `shows: []` as "no associated shows" (render a dash), not an error.
- Use `payment_status` for the Payment Status column; use raw `status` for lifecycle badges.
- Pay action: show only when `can_pay === true`. The pay destination is NOT provided by this endpoint (separate story).

## Implementation Order

1. **`orders.helpers.ts`** — pure derivations + `OrderShowDto`/`OrderPaymentStatus` types. No Nest/Prisma imports beyond `Prisma.Decimal` typing. Easiest to test first.
2. **`dto/index.ts`** — `OrderListQueryDto extends PaginationDto` (+ optional trimmed `search`), `OrderShowDto`, `OrderListItemDto`, `OrderListResponseDto` (`{ data, meta: PaginationMetaDto, isEmpty }`), all with `@ApiProperty`/`@ApiPropertyOptional`.

4. **`orders.controller.ts`** — `@Controller('orders')`, `GET /` with guard, swagger (`@ApiOperation` description as a `[...].join(' ')` array opening with `**[SBE-13.2 — Order Listing]**`), `@ApiOkWrappedResponse`, `@ApiAuthResponses`; delegates to service, wraps in `successApiResponse`.
5. **`orders.module.ts`** + register in **`app.module.ts`**.
6. **Specs** for helpers, service, controller.

### `orders.helpers.spec.ts`

**deriveOrderPaymentStatus** (4): unpaid when `paid_amount = 0`; partially_paid when `0 < paid < total`; paid_in_full when `paid >= total`; zero-total order treated as paid_in_full when `paid >= 0`.
**deriveIsOverdue** (4): true when a transaction is past `due_date` and `scheduled`/`processing`/`failed`; false when all `succeeded`; false when past-due but `canceled`/`refunded`; false when `due_date` is future.
**deriveCanPay** (3): true for `partially_paid`/`unpaid` active order; false for `paid_in_full`; false for `canceled`/`refunded` even if not fully paid.
**buildOrderShows**: single show; multi-show order deduped + sorted to distinct shows; null city / To-Be-Announced (null) date pass through; `[]` for an order with no `showProduct`. (No dedicated "default-included add-on" case — its show is contributed by the parent booth line in the same order and is already pinned by the dedup-by-show-id case; see D4.)

### `orders.service.spec.ts`

- meta correctness — `page/limit/total/totalPages/hasNext/hasPrev` for one / few / many orders (13.2-l).
- `where` scopes by resolved `company_id` + `order_type: product` + `deleted_at: null` — **`product`-only** (SBE-1621), no `status` narrowing (asserts `pending` product orders included).
- ordering `created_at desc`.
- `isEmpty === true` and `data: []` when count is 0 (13.2-m).
- each row maps derived `payment_status` / `is_overdue` / `can_pay` / `shows` correctly (caller-shape fidelity: feed a split product order + a subscription order in one page).
- ownership: unknown/inactive exhibitor → `resolveExhibitorCompanyId` throws `NotFoundException` (13.2-j).
- `search` adds the `order_number` contains predicate when provided; absent otherwise.

### `orders.controller.spec.ts`

- calls `service.listOrders(req.user.id, query)` and returns `successApiResponse('Orders retrieved successfully', data)`.
- query DTO defaults (`page=1`, `limit=10`) flow through.

## Security Considerations

- **Ownership enforcement (13.2-j):** `company_id` is derived server-side via `resolveExhibitorCompanyId(req.user.id)`; the client cannot pass a company/exhibitor id. Every query is filtered by it. A cross-tenant test asserts exhibitor B never sees exhibitor A's orders.
- **Auth:** class-level `JwtAuthGuard`; unauthenticated → 401 (documented by `@ApiAuthResponses`).
- **PII / billing exposure:** the listing returns only summary columns (no billing address, no card data, no Stripe ids). Monetary values are read straight from `Order` columns (no recomputation), satisfying 13.2-k.
- **Input hardening:** pagination bounded by `PaginationDto` (`@IsInt`, `@Min(1)`, `@Max(100)`); `search` trimmed + length-capped.

## Deferred

- **Invoice action / endpoint (13.2-g — `GET /orders/:orderId/invoice`) — DEFERRED pending team confirmation of invoice semantics** (Open Question #3 in the feasibility doc). Reason: invoices are persisted only on successful payment (unpaid → 0, split → N), and the only renderer (`ppl.service.renderInvoicePdf`, pdfkit) is subscription-shaped — there is no product-order renderer. The team must confirm **(A)** generate an order-level invoice/summary PDF from `Order`+`OrderItems` for every order, vs **(B)** serve only persisted `Invoice` rows. Until then this story ships **no** invoice endpoint and **no** Invoice action-availability flag in the row DTO. Re-scope once A vs B is decided.
- **`GET /orders/:orderId` order-details aggregate (13.2-f navigation target)** — owned by story 13.3; this listing only returns the `id` the View action navigates to.
- **Pay-flow execution (13.2-h2)** — out of scope; owned by Credit Card Payment Workflow / Payment Schedule Visibility stories. Only `can_pay` is emitted here.

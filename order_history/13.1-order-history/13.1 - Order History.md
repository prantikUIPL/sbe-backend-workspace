## Summary

Story 13.1 is the Order History container/page story. It introduces NO independent endpoint or logic — every requirement (list all of the exhibitor's past+current orders, own-orders ownership scoping + backend enforcement, completeness, empty-state, and order-level actions) is delivered by building story 13.2 (Order Listing Table), which owns the single canonical GET /orders listing endpoint. The one item formerly unique to 13.1 — the empty state — has been folded into 13.2 as 13.2-m, so focusing on 13.2 completes 13.1 automatically. All items are Deliverable on the built exhibitor listing primitives (company-scoped order listing, paginated meta with isEmpty/total); order-level actions (View/Invoice/Pay) are defined and built under 13.2.

## Open Questions

- **[OPEN — for team discussion]** Should abandoned 'pending' orders (a pending row is created at place-order time before the Stripe redirect; if the user backs out it stays pending forever) be shown in Order History, or excluded? The existing PPL order-history endpoint deliberately excludes them and shows only terminal/active states, but 13.1's mandate to show 'current' orders plus the sibling 'pay' action imply active payable orders (e.g. partially_paid split plans) must appear — confirm whether never-paid abandoned-checkout orders are included or filtered out.
  - **Current direction (2026-07-01):** mirrors Story 13.2 — the shipped `GET /orders` **includes `pending` rows** so in-flight/current orders surface (consistent with 13.1's "past AND current" mandate and the Pay action on payable orders). Flagged for team confirmation because this also surfaces never-paid abandoned-checkout rows; team to decide whether to keep, filter by an age cutoff, or exclude pure abandoned-checkout `pending` orders (a single `where` predicate if it changes).
